# 訪談分析、Laddering 與抽樣

## Laddering 手段目的鏈

結構：`attribute（屬性）→ consequence（結果）→ value（終端價值）`。受訪者說得出的通常只有屬性層，價值層要靠追問爬上去。

### 起手

抓一個受訪者自己提到、且明顯在意的屬性當梯子的第一階：

- 「你剛剛說你選它是因為『輕』——輕，對你來說差在哪？」

### Why 追問的變形話術

連環「為什麼」會有審問感，受訪者三次之後就開始防衛。直接的「為什麼那對你重要？」每三次追問最多用一次，其餘輪替：

- 結果化：「這件事做到了，會帶來什麼？」
- 負向 laddering：「假設它做不到／重一點，會發生什麼事？」（人對損失的描述比對獲得的更具體）
- 差異化：「跟你之前的做法比，最大的差別是什麼？那個差別為什麼有感？」
- 第三人稱軟化：「有些人完全不在意這個——你覺得你為什麼特別在意？」
- 情境回放：「上次它幫上忙是什麼時候？那當下你心裡想什麼？」

### 停止時機

- 到達終端價值就停：安全感、成就感、被認同、掌控感、自由、家人優先……這類詞出現代表到頂了。
- 受訪者開始重複、或說「就……很重要啊」＝梯子到頂，再追只會逼他編造。
- 不要硬爬到哲學層次。「不想遲到」爬到「守時＝自我認同」即可，不用爬到人生意義。

### 記錄格式

一條 ladder 一行，分析時可跨受訪者比對哪幾條鏈重複出現：

`輕(A) → 通勤不累(C) → 每天願意帶出門(C) → 覺得自己有把生活打理好(V) [P05:22-27]`

## 逐字稿分析流程

### 1. Open coding — 逐句貼標

- 單位：一個意思一個標籤。一句話裡有兩個意思就貼兩個標籤。
- 標籤用受訪者的語言（in-vivo coding），不要一開始就翻成理論詞。「怕買了又跟上次一樣吃灰」不要先壓成「損失規避」。
- 每個標籤附出處 `[P03:14]`（受訪者編號:段落），之後所有洞察靠這個追溯。
- 分軌標記：`行為`、`情緒`、`數字`（金額/時間/次數）、`四力`（push/pull/anxiety/habit）、`job`（功能/情感/社會）、`ladder`。
- 量的體感：一份 60 分鐘逐字稿正常產出 40-80 個標籤。低於 20 個通常是訪談太淺（都是意見沒有事件），這本身要回報。

### 2. Affinity mapping — 由下往上分組

- 順序是「先湊堆、再命名」。先設好類別往裡塞是驗證不是探索，會讓你只看到你想看的。
- 一堆 3-8 個標籤；超過 10 個代表這堆混了兩個概念，拆開。
- 跨受訪者才算 pattern：同一堆至少要有 2-3 位不同受訪者的標籤。單一受訪者講十次也只算一個人的執念。
- 落單的標籤保留在「未分組」區，不要硬塞——它們可能是下一個 segment 的第一個訊號。

### 3. 命名主題

- 主題名要是一個有張力的句子，不要單詞：「怕選錯，而不是怕貴」優於「價格敏感」；「想要的是交代得過去，不是最好」優於「B2B 決策心理」。
- 好的主題名讓沒讀逐字稿的人也能做決策。

### 4. 反例檢查

- 每個主題定案前，回逐字稿找 disconfirming evidence：誰說了相反的話？做了相反的事？
- 找到反例後問：這個人跟其他人差在哪（情境、角色、經驗值）？反例往往不是雜訊，是 segment 邊界的訊號。
- 找不到任何反例反而要警覺：可能是篩選偏差（招來的人太同質），在報告中註明。

### 編碼表格式

用表格管理標籤，之後洞察與 handoff 都從這張表長出來：

| 欄位 | 說明 |
|---|---|
| `tag` | 標籤（in-vivo 優先） |
| `type` | 行為 / 情緒 / 數字 / 四力 / job / ladder |
| `source` | `[P03:14]` 受訪者:段落 |
| `verbatim` | 原句摘錄（分析時引用的就是這格） |
| `evidence_grade` | S / A / B / C（見證據強度分級） |
| `group` | affinity 分組後回填 |

### 多人編碼的一致性

兩人以上分工編碼時：

- 先各自編同一份逐字稿，對出入處討論到共識（negotiated agreement），把裁決規則寫下來再分頭編其餘場次。
- 不追求量化的 inter-rater 係數——探索性研究做到「規則明確、爭議留痕」即可，別把時間花在算 kappa。
- 中途新增的標籤類別要回頭掃已編完的場次。

## 洞察句式模板

`[某類人] 在 [情境] 時想要 [動機/progress]，但 [阻礙]，所以 [現行解法與代價]。`

範例：`剛接手部門報表的新手主管，在月底要向上匯報時想要一份不會被挑戰的數字，但不信任自己拉的樞紐分析，所以每次都拜託前任留下的同事幫忙重看一遍，欠下人情且晚兩天交。`

規則：

- 五個成分每一格都要能指回引文出處，指不回去的那格就是你的腦補。
- 「某類人」用行為定義（剛接手報表的、每週出差三天的），不用人口屬性。
- 「現行解法與代價」不能空白。找不到現行解法通常代表問題不夠痛——人對真正的痛一定已經有土法煉鋼。
- 每條洞察標註：證據等級（見下）、支持人數（n=5/12）、代表引文 1-2 條。

### 從洞察到決策的 so-what 檢查

每條洞察定稿前過三關，過不了的降級為觀察，不進結論：

1. 這條洞察如果為真，我們會做什麼不一樣的決定？答不出來＝沒有決策價值。
2. 這條洞察如果為假，最可能是哪裡錯了？（樣本偏差、引導式提問、單一場次過度放大）
3. 要再多確定一點，最便宜的下一步驗證是什麼？（多訪 2 場該 segment、跑一個 landing page 測試、翻交易紀錄）

## 證據強度分級

| 等級 | 內容 | 例子 |
|---|---|---|
| S（行為＋金錢） | 已付費、預付、簽 LOI | 「他上季付了 3 萬找外包做這件事」 |
| A（行為） | 實際做過的事、自建 workaround、換過方案、投入過具體時間 | 「他自己維護一份 200 行的 Excel 巨集」 |
| B（commitment） | 答應具體下次會議、引薦他人、試用並回饋 | 「當場約了下週給團隊 demo」 |
| C（口頭意見） | 說法、偏好、「我覺得」「我會」 | 「他說這功能很棒一定會用」 |

規則：C 級只能當假設線索，不得單獨支撐任何結論；每條洞察至少要一條 A 級以上證據。報告中每條洞察的證據組成（幾條 S/A/B/C）要列出來。

## 樣本數與理論飽和

- 停止規則：連續 2-3 場訪談沒有產生任何新編碼、新主題 → 該 segment 飽和，可以停。
- 常見區間：B2C 8-15 場；B2B 5-10 場，但必須涵蓋角色——用戶、買家、決策者各至少 1-2 位，只訪用戶不算飽和。
- 飽和是按 segment 各自計算，不是總場次。訪到第 6 場發現一個行為明顯不同的新群體，這個新 segment 要重新起算。
- 第 3 場之後就開始編碼，邊訪邊分析。理由有二：飽和只有邊分析才判斷得出來；前幾場會暴露大綱的爛問題，要趁早改。
- 大綱中途修改要記錄版本，分析時註明哪些場次用哪版大綱。

## 招募篩選

### 篩行為，不篩人口屬性

- 「過去 90 天內實際做過 X」優於「25-34 歲女性」。買行為的是情境與 job，不是年齡。
- 篩選條件直接對齊研究問題：研究「怎麼選記帳工具」就篩「過去半年換過或認真評估過記帳工具的人」。

### 篩題設計，避免自我報告偏差

- 不要讓正確答案太明顯。問「你重視健康嗎」人人說重視；改問行為回憶題：「上週你自己開伙幾次？最近一次是哪天、煮了什麼？」答不出細節的剔除。
- 加誘餌選項抓亂答者：選項清單裡混入一個不存在的產品或服務，勾了的直接剔除。
- 頻率題給具體區間（0 次／1-2 次／3 次以上），不給「經常／偶爾」這種各自表述的詞。

### 排除職業受訪者

- 篩題加一條：「過去六個月你參加過幾次有酬訪談或市調？」超過 2 次剔除。
- 社群徵才貼文不要寫明訪談主題細節與報酬金額，減少為了報酬而自我包裝的人。
- B2B 的 switch 樣本從自己的資料找最準：剛成交客戶、剛流失客戶、銷售輸單名單。

### 招募節奏

- 用 rolling 方式排程：一次只約 3-5 場，分析完再約下一批。一口氣約滿 15 場等於放棄用飽和判斷省時間，也讓爛篩選器一路錯到底。
- 每批之間允許修篩選器：發現訪到的人沒有目標行為，是篩題壞了，先修再繼續。
- 每場之間留 30-60 分鐘寫 debrief：三個最意外的點、要改的題目、初步標籤。當天不寫，一週後就只剩印象。

## 誘因設計與利益衝突

- 誘因的定位是「感謝你的時間」，不是「買你的答案」。金額對齊受訪者的時薪水準即可；過高會吸來職業受訪者，也會讓回答變客氣。
- B2B 中高階受訪者常不適合給現金：改用指定捐款、研究結果摘要回饋、或定位成 peer exchange（「想跟你請教業內做法」）。
- 利益衝突名單，這些人的讚美打一折：朋友與家人、你的下屬、想維持關係的現有大客戶窗口、想賣你東西的人。不是不能訪，是編碼時標記關係並降級口頭證據。
- 訪談者同時是創辦人時：開場明講「今天講缺點對我幫助最大，講好話對我沒用」；分析時把自己主持的場次標記出來，與第三方主持的場次比對讚美比例，做偏差檢查。
