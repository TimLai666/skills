# 原則、筆記類型與慣例

## 六原則詳解

### 1. 原子性（Atomicity）

一張卡只裝一個想法。原子性的單位是「想法」，不是句子也不是主題。
一個想法可以是一個主張、一個機制、一個定義、一個判準，通常 3-10 句話能講完。

為什麼：只有原子卡片能被自由重組。一張卡混了兩個想法，之後任何脈絡要引用其中
一個，都得拖著另一個，連結的精準度就毀了。

檢驗法：用一句陳述句當標題。寫不出來，或必須用「與、以及、和」串接，就是多個想法。
實體卡的單位是「實體」，檢驗法改為：一個實體名稱涵蓋得住卡內全部內容。

### 2. 自主性（Autonomy）

每張卡拿出來單獨讀要能成立，不依賴來源脈絡、不依賴「上一張卡」。
需要的背景就在卡內用一兩句話交代，或連結到解釋背景的卡。

為什麼：卡片會在你意想不到的脈絡被重新讀到。寫的當下省略的背景，半年後就是斷片。

### 3. 自己的話（Own Words）

永久卡必須改寫，不能貼原文。摘抄只證明你看過，改寫才證明你懂了。
原文屬於文獻筆記層，永久卡引用文獻筆記而不是直接引用原始來源。

### 4. 連結優先（Connection over Collection）

連結長在內文。內文提到另一張卡的概念或實體時，就地把那個詞變成連結，
必要時用 `[[目標卡|顯示文字]]` 讓語句保持通順；能讓句子直接容納完整標題
就不用別名，別名是次選。關係由句子本身說明：

```markdown
間隔重複有效的原因是每次複習都迫使大腦執行一次真正的提取，而
[[提取練習比重讀更能鞏固記憶|提取本身就是強化]]。
```

只有一種情況用文末的「相關卡片」補充段：這張卡與另一張卡關係重要，
但內文沒有自然提到它（常見於對比與反駁）。這時才列點，條目格式固定為
「關係標籤：[[完整標題]] — 一句話說明」：

```markdown
## 相關卡片
- 反駁：[[熟悉感是有效的學習訊號]] — 本卡主張熟悉感會誤導
```

為什麼：卡片盒的檢索靠連結網絡，不靠資料夾與搜尋。為了連結而另立清單，
連結就跟內容脫節；長在內文的連結才記錄了「在什麼脈絡下想到這張卡」。
結構筆記不受此限，它天生就是連結清單。

### 5. 湧現結構（Emergent Structure）

不要先設計分類樹再往裡塞卡。結構筆記（hub）是事後整理出來的地圖，
隨著某個主題的卡片變多才建立，並隨連結成長更新。

### 6. 矛盾是發展點

新想法與既有卡衝突時，三不一要：不覆蓋、不刪除、不硬湊，要寫成新卡，
與被衝突的卡互連並標明分歧點。矛盾對是最有價值的思考素材。

## 筆記類型詳解

### 靈感筆記（fleeting）

隨手記，不求格式。唯一規則：1-2 天內處理掉，煉成永久卡或丟棄。
inbox 積壓超過一週就是系統警訊。

### 文獻筆記（literature）

一個來源一份，記「這個來源說了什麼、哪裡值得回頭引用」。用自己的話、極簡、
必附出處（書頁、URL、章節）。它是永久卡的原料倉，不是永久卡本身。

### 永久筆記（permanent）

系統的本體。自己的話、至少 1-2 條說明關係的連結、`sources` 指向文獻筆記。分兩種：

- **概念卡** — 一個主張、機制、定義或判準。陳述句標題，一卡一想法。
- **實體卡** — 一個人、事件、組織、作品或事物（如一場颶風）。實體名稱標題，
  一卡一實體，內容是事實、時間線或操作步驟，而非論點。具名方法／流程／SOP
  也是實體卡（步驟寫成可掃讀清單）；方法是否有效、適用邊界則是概念卡的事。
  實體卡透過連結掛上概念卡：
  實體是概念的實例，概念靠實體累積證據。百科式的敘事段落要改寫成可掃讀的
  結構（概要＋時間線＋關係），不要整段照搬。

### 結構筆記（structure）

某個主題的入口與地圖：列出該主題的卡片、安排閱讀動線、標出空缺。
結構筆記可以互相連結形成更高層的地圖。總入口是 `index.md`。

## 命名與 frontmatter 慣例

- 檔名：標題直接當檔名，概念卡是陳述句（`間隔重複比集中複習有效.md`）、實體卡
  是實體名稱（`颶風基科-1989.md`）。中文內容用中文標題，英文內容用小寫連字號
  slug（`spaced-repetition-beats-cramming.md`）。
- Luhmann 式時間戳 ID（`202607181530-`）為可選前綴，使用者沒要求就不加。
- 既有筆記庫仍用舊鍵 `source:` 時，視為現場慣例沿用即可；audit 時提示統一，不判違規。
- 概念卡主張被修正而需改標題時：改名、全庫更新入連結、逐一檢查顯示別名是否
  過期，一次做完並記錄。
- 若筆記庫將對外發佈為網頁（URL 管線），改用 ASCII slug 檔名、中文標題放
  `title` 欄——中文檔名的編碼成本只在發佈情境落地。
- frontmatter 最小集：

```yaml
---
title: 陳述句標題（實體卡：實體名稱）
type: fleeting | literature | permanent | structure
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources: [literature/某文獻筆記.md]   # 一律清單。permanent 卡指向文獻筆記；literature 卡放原始出處
---
```

## 卡片範例

四張完整範例：文獻筆記、概念卡、實體卡、結構筆記。前兩者與結構筆記
共用「學習方法」主題串成小型網絡，實體卡示範百科式內容的改寫。

### 文獻筆記：`literature/ahrens-2017-how-to-take-smart-notes.md`

```markdown
---
title: Ahrens《How to Take Smart Notes》重點
type: literature
created: 2026-07-18
updated: 2026-07-18
sources: ["Sönke Ahrens, How to Take Smart Notes (2017)"]
---

- 筆記系統的瓶頸不在收集而在思考的外部化，寫作是研究的媒介不是產出。（ch.1）
- 提取練習比重讀更能鞏固記憶，間隔安排讓每次提取保持適度困難。（ch.10）
- 分類樹會過早固定思考方向，連結讓主題自然聚集。（ch.7）
```

重點：極簡、自己的話、每條標章節。它是原料倉，觀點的展開留給永久卡。

### 概念卡：`permanent/間隔重複比集中複習有效.md`

```markdown
---
title: 間隔重複比集中複習有效
type: permanent
created: 2026-07-18
updated: 2026-07-18
sources: [literature/ahrens-2017-how-to-take-smart-notes.md]
---

集中複習靠重讀製造熟悉感，但熟悉感不等於記得。間隔重複有效的原因是每次複習都
落在快要忘記的時候，迫使大腦執行一次真正的提取，而
[[提取練習比重讀更能鞏固記憶|提取本身就是強化]]。代價是這套方法
[[間隔重複只適合原子化知識點|只適合原子化的知識點]]，需要整體理解的概念不適用。
```

重點：陳述句標題、一個想法講完就停、連結就地長在內文提到的詞上（機制與適用
邊界都在句子裡連出去，不另立清單）、`sources` 指向文獻筆記。

### 實體卡：`permanent/颶風基科-1989.md`

原始素材是一段百科式敘事（「颶風基科是 1989 年太平洋颶風季的第 11 個獲得命名的
風暴。基科於 8 月 25 日從一片大型中尺度對流系統形成，之後緩慢向西北方向移動，
於次日早上快速增強並達到颶風強度……」），改寫成實體卡：

```markdown
---
title: 颶風基科（1989）
type: permanent
created: 2026-07-18
updated: 2026-07-18
sources: [literature/wikipedia-hurricane-kiko-1989.md]
---

[[1989年太平洋颶風季]]第 11 個獲命名的風暴。由大型中尺度對流系統形成，
兩天內增強至最高強度並在下加利福尼亞州南端登陸。

## 時間線
- 08-25 形成，緩慢往西北移動
- 08-26 早上[[熱帶氣旋的快速增強|快速增強]]，達颶風強度
- 08-27 清晨達最高強度（風速 195 km/h）
- 08-27 [[世界協調時間|UTC]] 06:00 左右於蓬塔阿雷納附近登陸
```

重點：實體名稱標題（不硬套陳述句）、一卡一實體、敘事段落改寫成概要＋時間線的
可掃讀結構。連結就地長在內文提到的詞上：颶風季、快速增強、UTC 各自在出現處
連出去，跟維基百科的行文方式一樣，不另立連結清單。

### 結構筆記：`structure/學習方法.md`

```markdown
---
title: 學習方法
type: structure
created: 2026-07-18
updated: 2026-07-18
---

記憶與練習相關卡片的入口。

## 記憶機制
- [[提取練習比重讀更能鞏固記憶]]
- [[間隔重複比集中複習有效]]
- [[間隔重複只適合原子化知識點]]

## 工具
- [[anki]]

## 空缺
- 尚無卡片：交錯練習（interleaving）與間隔重複的關係
```

重點：只做地圖不裝內容，動線分組隨卡片成長調整，「空缺」段標出下一步該補的卡。

## 預設資料夾結構（僅在新建時使用）

```
slipbox/
├── index.md          # 總入口結構筆記
├── inbox/            # 靈感筆記（待處理）
├── literature/       # 文獻筆記
├── permanent/        # 永久筆記
└── structure/        # 結構筆記
```

在 Obsidian 中此目錄可直接作為 vault，`[[wikilink]]`、graph view 均可用。
語法細節參考 `obsidian-markdown` skill。

## 套用在既有知識庫

紀律可以套在任何結構上。原則：沿用該庫的目錄、frontmatter 與連結語法，
只引入原子性、拆卡、連結三項紀律與變更檢核清單。

### llm-wiki 對應表

| llm-wiki 元素 | 卡片盒角色 | 紀律適用 |
|---------------|-----------|----------|
| concept 頁 / entity 頁 | 概念卡 / 實體卡 | 完整適用：原子性、拆卡、連結檢核 |
| 全域與 topic `index.md` | 結構筆記 | 導航用，不套原子性 |
| comparison / query 頁 | 綜合筆記 | 本質是多卡合成，不套原子性，但連結紀律照跑 |
| `raw/` 來源檔 | 文獻筆記層 | 不可變；wiki 頁用自己的話改寫並標 provenance |
| `log.md` | 無對應 | llm-wiki 自有機制，照舊 |

對應表補充三則：

- raw/ 是逐字原文，不是「自己的話」的文獻筆記。wiki 情境中「自己的話」義務由
  wiki 頁承擔，wiki 頁 `sources:` 直指 raw/ 屬合規，不受「永久卡引用文獻筆記、
  不直接引用原始來源」規則約束。
- 靈感筆記（fleeting）無 wiki 對應：llm-wiki 不設 inbox，未加工素材直接進
  raw/ 或不落檔。
- 建頁門檻：wiki 域內 llm-wiki 的 Page Thresholds（2+ 來源或單來源核心才建頁）
  優先於卡片盒「有想法就建卡」——這是 wiki 防 passing-mention 噪音的刻意覆寫。
- 矛盾標記：正典的措辭是「依該庫慣例標記分歧」；llm-wiki 的具體慣例是
  contradictions／contested frontmatter＋Update Policy。這是刻意的分層設計，
  不是漂移。

llm-wiki 的頁面寫入流程如何掛上檢核清單，見該 skill 的 Note Discipline 一節。

### llm-wiki 情境範例：ingest 觸發拆頁

ingest 一篇新文章時，要把「訓練成本隨參數量成長的估算」加進
`concepts/transformer-architecture.md`。寫入前跑檢核清單：

1. 原子性：該頁主題是「Transformer 架構是什麼」，訓練成本是另一個問題 →
   命中多問題測試。
2. 拆卡：新開 `concepts/transformer-training-cost.md`。新頁內文提到架構的
   地方就地連結 [[transformer-architecture]]；原頁在提到訓練成本的那句話
   加上指向新頁的連結。
3. 收尾（llm-wiki 自有機制照跑）：新頁補 frontmatter 與 provenance、
   加進 topic index、更新既有頁的入連結、追加 log.md。

反例：同一篇文章補充了 attention 機制的更清楚解釋 → 同一個想法的深化，
直接更新 `transformer-architecture.md` 並 bump `updated`，不開新頁。
