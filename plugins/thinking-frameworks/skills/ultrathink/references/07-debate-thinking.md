# 辯論思考

## 主題定位

辯論賽制把爭議攤開成有結構的攻防：先立好戰場（爭點或判準、定義），讓對立觀點在同一個戰場上各自舉證、互相攻擊，最後依標準裁決，不憑氣勢或立場。這比單方面自我檢查更能揪出盲點——自己想「有沒有問題」容易放水，被要求正面攻擊時才會真的用力。

兩個工具對應辯論的兩大辯題類型：政策性辯題（該不該做某個改變）用需根解損，價值性辯題（某個立場或主張成不成立）用判定立駁。前面主題產出的方向、方案、主張，到這裡全部當成待裁決的辯題壓一輪。

原方法都是多辯士賽制。Agent 使用時，優先用 subagent 還原多辯士結構，各角色的隔離方式依工具而異，見各工具的操作步驟；環境不支援 subagent 時，才退回單人依序分飾。

## 論點三要素：主張、資料、推論

不論用哪個工具，單一論點的建立都離不開三要素（源自圖爾敏論證模型的核心）。這不是獨立跑一輪的工具，而是本主題所有論點的建構與攻擊標準：需根解損的四關舉證、判定立駁的立論與反駁，每個論點都用它構建、也用它檢驗。

| 要素 | 內容 |
| --- | --- |
| 主張（Claim） | 你想證明什麼？一句可以被反對的斷言 |
| 資料（Data） | 支持主張的客觀數據、專家意見或歷史案例 |
| 推論（Warrant） | 透過邏輯說明「為什麼」這些資料能夠證明主張 |

建構時三要素缺一不可：缺資料是空口主張，缺推論是資料堆砌——資料再多，沒說明為何支持主張就不成論證。推論最容易被省略，因為它常被當成「顯而易見」；但隱含的推論正是論點最脆弱的地方，必須明寫出來。例如主張「預設開啟通知侵蝕信任」、資料「通知疲勞導致全關的使用者行為數據」，推論是「未經同意的打擾讓使用者感到失控，失控感直接損害信任」——把這句寫出來，才看得見它依賴「使用者感到失控」這個可被檢驗的環節。三要素怎麼攻，見下方質詢拆解。

## 質詢拆解：三個提問維度，檢驗任何論點

質詢是辯論的攻擊技術，同一套提問拿來檢驗對方的論點，也拿來先掃自己的破綻。三個維度中，前兩問正好對應論點三要素的資料與推論，第三問則是三要素照顧不到的出口：

| 維度 | 提問 | 攻擊面 |
| --- | --- | --- |
| 事實是否為真？ | 引用的數據、歷史案例或因果關係，有可信來源支持嗎？ | 攻資料：舉證不實、樣本偏差、資料過時 |
| 邏輯是否嚴密？ | 推導過程有沒有滑坡謬誤、偷換概念、稻草人謬誤？ | 攻推論：謬誤、斷鏈、資料到主張之間跳步 |
| 有無其他可能？ | 就算現象描述為真，導出的是唯一的結論或方案嗎？ | 攻唯一性：找替代解釋與替代方案 |

打法上的優先序：直接駁主張最弱，通常只剩「與判準無關」一條路；最有效率的攻擊點往往是對手沒寫出來的隱含推論；而第三問是質詢的獨門武器——前兩問攻論點本身，第三問承認論點卻攻它的唯一性：「你說的都對，但為什麼是這個結論？」現象為真不代表這個解釋是唯一解釋，問題成立不代表這個方案是唯一方案。

用在本主題的兩個工具：判定立駁的駁論階段照這三個維度打；需根解損的四關檢驗本質上就內建了它——根問「有無其他成因」、解問「有無繞過路徑」，都是第三問的化身，四關 subagent 舉證時可用三個維度自問補強。任何一方立論或舉證寫完，先用同一套提問掃自己，會被自己問倒的論點不要端出去。

## 需根解損：四關逐一舉證，裁決變更提案是否成立

政策性辯論用四大爭點檢驗一個改變現狀的提案：需要性、根屬性、解決力、損益比。正方要讓提案成立，四關都得舉證過關；任何一關倒下，提案就不成立。套用到任務上，任何「我打算這樣做」都可以重述成一個變更提案來檢——功能變更、流程改動、重構、debug 的修法、文案的切角都適用。一句話：**四關逐一舉證，提案不成立就不動手。**

| 關卡 | 檢驗問題 |
| --- | --- |
| 需（需要性） | 現狀是否有問題，或有可獲得的利益？有沒有改變的理由？ |
| 根（根屬性） | 問題是否根植於現行制度或做法本身？提案是否對症下藥？ |
| 解（解決力） | 提案實施後，真的能有效解決問題、實現利益嗎？ |
| 損（損益比） | 提案帶來的利益，是否大於衍生成本與負面影響？ |

跟前面主題的分工：權衡得失的決策矩陣比較的是「改成哪個」，需根解損檢驗的是「該不該改」；PMI 與力場分析找全因素、看清推拉力，需根解損拿這些當證據做成立性總裁決。預測未來的推演正好餵進解與損——方案的長期效果與衍生成本，用模型推比用印象猜可靠。

四個爭點在賽制裡由不同辯士分頭攻防。Agent 使用時，優先用 subagent 還原這個結構：四關各開一個獨立 context 的 subagent，每個只拿到現狀描述、提案內容與自己那一關的檢驗問題，看不到其他關的結論，立場才不會互相帶風向。主 agent 負責統整裁決。環境不支援 subagent 時，才退回單人依序分飾四關。

### 操作步驟

1. **寫下現狀與提案** — 一段現狀描述（含已知事實與依據）、一段提案內容（要改什麼、怎麼改）。這是四個 subagent 共用的唯一輸入，措辭保持中立，不夾帶支持或反對的立場。
2. **平行開四關 subagent** — 每個 subagent 的提示只含：現狀、提案、該關的檢驗問題，並要求每個論點照三要素構建（主張、資料、推論）、標明假設、給出該關成立與否的結論。四關互不相見。
   - **需**：找現狀的實際問題或未實現的利益，每一條附依據；找不到就誠實回報「無改變理由」，不硬湊。
   - **根**：追問題的成因是否出在現行制度或做法本身。若問題來自執行不力、外部因素或一次性事件，改做法就不對症；同時檢查提案打的是不是那個成因。
   - **解**：推演提案實施後的因果鏈，逐環檢查是否成立、依賴了哪些未驗證的假設、有沒有繞過方案的路徑——能被規避的方案，解決力存疑。
   - **損**：盤點提案的衍生成本與負面影響（實施成本、轉換成本、新風險、副作用、機會成本），與預期利益比較。
3. **依序裁決（主 agent）** — 回收四關結論，照 需→根→解→損 的順序裁決。這是舉證鏈：前一關不成立，後面的結論只當參考，不能拿「損益比划算」救一個根本不需要或不對症的提案。
4. **給出裁決與修補方向** — 四關全過：提案成立，把各關標出的假設列為後續驗證項。任何一關倒下：指出倒在哪一關、為什麼，並提出修補方向（縮小提案、換打點、先補驗證），不是只說不行。

### 對照範例：該新增積分獎勵系統嗎？

現狀：產品月流失率 40%，高於同類產品。提案：新增積分獎勵系統，提高使用者留存誘因。

| 關卡 | 檢驗結果 | 裁決 |
| --- | --- | --- |
| 需 | 流失率 40% 有數據佐證，確有改變的理由 | 過 |
| 根 | 使用者訪談顯示流失主因是核心功能載入過慢，不是缺乏誘因——提案沒打中成因 | **倒** |
| 解 | （參考）就算做了，對載入慢造成的流失沒有解決力 | — |
| 損 | （參考）開發兩個月＋長期維運成本 | — |

裁決：倒在根屬性。解與損的結論只當參考，不回頭用「積分系統成本不高」翻案。修補方向：提案改成效能優化；積分系統留待流失主因排除後再評估。

### 常見錯誤

- 只檢解與損，跳過需與根：直接辯「方案好不好」，沒先確認「該不該改」。多數不成立的提案倒在前兩關。
- 四關看得到彼此的結論：解的樂觀推演帶壞損的成本盤點，需的問題清單引導根往制度面找——立場互相污染，四關變一言堂。
- 把「有問題」當成「需要這個提案」：需只證明改變的理由，不證明這個提案；對不對症是根的事，有沒有效是解的事。
- 根屬性只問「有沒有問題」不問「成因在哪」：執行面的問題被拿來當改制度的理由。
- 拿後面的關卡救前面：損益比再划算，也救不了不需要或不對症的提案。
- 裁決只說不行不給修補方向：這是動手前的思考工具，目的是把提案修對，不是純否決。
- 現狀與提案描述夾帶立場：輸入偏了，四關全偏。

### 自檢清單

做完需根解損後逐項檢查，任何一項不成立就回去補：

- [ ] 現狀與提案寫成中立描述，沒有夾帶支持或反對的措辭
- [ ] 四關各自獨立檢驗：subagent 互不相見，單人分飾時沒讓前一關的結論帶風向
- [ ] 各關論點三要素齊備：有主張、有資料、推論有明寫，不是空口斷言或資料堆砌
- [ ] 需：改變的理由有依據，不是印象
- [ ] 根：問題成因追到了，且確認提案打的就是那個成因
- [ ] 解：因果鏈逐環檢過，依賴的假設有標出來
- [ ] 損：衍生成本與負面影響盤點過，有跟預期利益比較
- [ ] 裁決照 需→根→解→損 順序，沒有拿後面的關卡救前面
- [ ] 不成立時有指出倒在哪一關與修補方向

## 判定立駁：立好判準與定義，正反攻防後依判準裁決

價值性辯論探討抽象主張——「某做法是否正當」「某原則是否該優先」這類沒有客觀量尺的爭議，口訣是判、定、立、駁：先立判準與定義劃定戰場，再讓正反雙方立論攻防，最後依判準裁決。套用到任務上，凡是「該不該、好不好、符不符合某原則」的規範性判斷都適用——設計原則衝突、文案立場、命名取捨、預設值的正當性、規範與慣例的爭議。一句話：**沒有判準就沒有裁決，先立戰場再開打。**

| 步驟 | 內容 |
| --- | --- |
| 判（判準） | 評斷這場爭議勝負的最高標準或核心價值，例如「使用者信任」「長期維護性」 |
| 定（定義） | 對辯題中的關鍵字詞明確界定，統一戰場範圍，雙方講的必須是同一個東西 |
| 立（立論） | 提出支持己方立場的論點，以數據、邏輯或案例佐證 |
| 駁（反駁） | 指出對方論點中的邏輯謬誤、舉證不實或推論瑕疵 |

跟需根解損的分工：需根解損裁決政策性提案（該不該做某個改變），爭的是事實與因果；判定立駁裁決價值性主張（某立場成不成立），爭的是價值與原則。一個提案的事實層過了需根解損，價值層還有爭議時，接著用判定立駁。

賽制裡立論與反駁由對立的兩方辯士執行。Agent 使用時，優先用 subagent 還原攻防：判與定由主 agent 先做——判準和定義是雙方共用的戰場邊界，必須中立；立論階段正反各開一個獨立 context 的 subagent，互不相見，避免先立者帶風向；反駁階段交叉互駁，把對方的立論交給彼此——注意這裡跟需根解損的隔離不同，駁論必須看到對方論點才駁得動，隔離的是立論階段，不是反駁階段。主 agent 最後依判準裁決。環境不支援 subagent 時，才退回單人依序分飾正反雙方。

### 操作步驟

1. **定辯題（主 agent）** — 把爭議寫成一句可以站正反兩方的命題：「新功能預設替使用者開啟通知，是正當的。」寫不出對立面的命題不是辯題，是既定事實或個人偏好，不必開打。
2. **判：立判準（主 agent）** — 定出這場爭議的最高裁決標準，通常是一個核心價值或目標。判準必須中立於雙方立場——把判準定成己方立場的同義詞是循環論證。多個價值衝突時，明說以哪個為最高、為什麼。
3. **定：下定義（主 agent）** — 把辯題裡的關鍵字逐一界定清楚，包含範圍與排除。定義是給雙方共用的，不偏袒任何一方。
4. **立：正反立論（subagent）** — 正反各開一個 subagent，各拿到：辯題、判準、定義、己方立場。要求每個論點照三要素構建：主張明確、資料佐證、推論明寫，並明說論點如何連回判準。互不相見。
5. **駁：交叉反駁（subagent）** — 把正方立論交給反方駁、反方立論交給正方駁。只攻論證，不攻立場，照質詢拆解的三個維度打（見上）：事實是否為真、邏輯是否嚴密、有無其他可能；隱含的推論是最常見的突破口。駁不動的論點要明說駁不動。
6. **裁決（主 agent）** — 逐一清點：哪些論點被駁倒、哪些站住了，站住的論點在判準下各值多少。依判準裁決哪方成立，不數論點數量、不憑氣勢。被駁論揭示的問題，整理成修補方向或後續驗證項。

### 對照範例：預設開啟通知是否正當？

辯題：新功能預設替使用者開啟通知，是正當的。判準：長期使用者信任。定義：「預設開啟」＝安裝後未經詢問即開啟、可於設定關閉；「正當」＝在判準下利大於弊且尊重使用者。

| 階段 | 正方 | 反方 |
| --- | --- | --- |
| 立 | 重要訊息不漏接；多數使用者不改預設，預設值即產品的照顧；業界普遍如此 | 未經同意的打擾侵蝕信任；通知疲勞導致全關甚至解除安裝；慣例不等於正當 |
| 駁 | 駁「打擾侵蝕信任」：若通知內容確實高價值，打擾不成立——關鍵在通知品質，不在預設值 | 駁「多數不改預設」：不改不代表同意，可能是不知道能改，訴諸沉默；駁「業界普遍」：訴諸多數，與判準無關 |

裁決：反方成立。正方站住的論點只剩「重要訊息不漏接」，且依賴「通知都高價值」的未驗證假設；反方兩個主論點在「長期使用者信任」判準下都站住。駁論同時揭示修補方向：改為首次使用時詢問，並依通知類別分級——這比單純「不做」更能同時保住兩方在乎的價值。

### 常見錯誤

- 沒立判準就開打：雙方各自訴諸不同價值，各說各話，最後只能憑感覺選邊。
- 判準定成己方立場：例如辯「該不該重視隱私」卻把判準定成「隱私保障」，循環論證，開打前就內定了。
- 定義沒統一：雙方講的「使用者」「預設」根本不是同一個東西，整場攻防打空。
- 立論缺三要素：只有主張沒有資料，駁方一句「查無實據」就倒；只有資料沒有推論，是資料堆砌不是論證。
- 立論階段互相看得到：後立論的一方變成只在回應對方，自己的最強論點沒展開。
- 駁論攻立場不攻論證：「你就是不懂使用者」不是反駁；反駁只處理質詢拆解的三個維度——事實、邏輯、其他可能。
- 裁決數論點數量：三個弱論點不敵一個扣緊判準的強論點。回判準裁，不數數。
- 把價值爭議硬塞給需根解損：四關檢的是事實與因果，價值衝突在那裡會被化約成成本效益，該用判定立駁的戰場。

### 自檢清單

做完判定立駁後逐項檢查，任何一項不成立就回去補：

- [ ] 辯題寫成可以站正反兩方的命題
- [ ] 判準中立於雙方立場，多個價值衝突時有明說以哪個為最高、為什麼
- [ ] 關鍵字定義統一，雙方在同一個戰場上交鋒
- [ ] 立論每個論點三要素齊備（主張、資料、推論明寫），並連回判準
- [ ] 立論階段正反互不相見，反駁階段才交叉
