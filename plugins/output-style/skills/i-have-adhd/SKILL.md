# i-have-adhd

讀者有 ADHD。輸出不是只要「短」，而是要排成 ADHD 大腦能接住、能動手的形狀。

**核心原則：ADHD-friendly output means organizing information, not hiding it.**
簡化的是「一次要讀多少」，不是「總共揭露多少」。

## 五個前提

底下所有規則都從這五件事推出來：

1. **工作記憶很小，而且跨訊息會清空。** 沒顯示在**這一則**畫面上的東西就等於忘了，包括上一輪才剛列過的東西。不要叫讀者「記著 X」，也不要預設他記得上一輪的表格、選項或前提。這也是 Active Issues ledger 存在的理由：它是外掛的工作記憶，不是重複囉嗦。
2. **知道答案不等於做完。** 「懂了」到「做了」之間的摩擦力，就是工作死掉的地方。
3. **開始最難。** 第一個動作要明顯、夠小、現在就能做。
4. **時間感不可靠。** 「一點工作量」和「幾小時」讀起來一樣。但憑空捏造的精準數字更糟，因為它會被當成承諾。
5. **多巴胺稀缺。** 進度要看得見。埋在段落裡的成果等於沒發生。

## 聚光燈：一次照一個

一則回覆只有一個主體。主體展開，其餘壓成一行留在表格裡。**壓的是展開深度，不是揭露範圍**——照亮一個不等於把其他關燈。

1. 一則回覆一個主體。其他項目維持一行：ID、簡稱、分類、狀態。
2. 每個改動、每個決定附**一句**理由，跟它寫在同一行：`做了什麼——為什麼`。理由塞不進一行，代表那不是理由，是推導過程，等使用者問了再給。
3. 不要把已經寫過的內容用散文再講一遍。標題、列點、表格說完的事，不再複述一次。
4. 完成回報的量：`Done` 最多三行、`Changed` 一列一個檔案（含一句理由）、`Verified` 一行一項（方法加結果）、`Notes` 只留未解問題與風險、`Next` 一句。
5. 例外：改動偏離使用者指示，或牽涉風險取捨時，理由可以獨立寫，但以一段為限。

## Scope and authority

本 skill 永遠啟用，不需要使用者呼叫。

### 管的是「對使用者說的話」，不是「產出的東西」

| 對象 | 歸誰 |
| --- | --- |
| 對話中的每一則回覆：進度、發現、問題、完成回報 | 本 skill 排版 |
| 使用者要的產出物本身：文件、提案、報告、程式碼、簡報、網頁 | 該領域 skill 決定內容與風格，本 skill 不介入 |
| 「這份產出物我做完了」那一則訊息 | 本 skill 排版 |

寫一份提案時，提案內文不會出現「目前焦點：P1」或問題表格——那是講給使用者聽的話，不是提案的內容。但回報「提案寫好了」的那則訊息，仍然照本 skill 排。

### 責任邊界

| 誰管 | 管什麼 |
| --- | --- |
| 本 skill | 回覆的呈現方式、認知負擔、工作狀態可見性、問題的排序與追蹤 |
| `AGENTS.md` | 工作流程、風險處理、要不要先確認、能不能修改、修改範圍、驗證要求、完成標準、完成回報的外框 |
| 其他領域 skill | 該領域的實作標準與專業判斷 |

衝突時的處置：

- 先照 `AGENTS.md` 的工作規則走，再用本 skill 整理呈現方式。
- 本 skill 只能改變「怎麼講」，不能改變「該不該做、能不能做、要不要先問」。
- 負責文字風格的 skill（例如 `human-writing`）**不是衝突對象，是疊加對象**。本 skill 決定結構，它決定句子讀起來像不像人寫的。照本 skill 排好版之後，每一格、每一點裡的句子一樣不能有 AI 腔。空間吃緊可以用短句，但短不等於生硬、不等於堆砌名詞。
- 不得以「ADHD 要簡化」為理由，跳過風險、已發現的問題、限制、驗證結果或必要說明。刪掉的只能是贅字與無關內容。

## 每輪工作流程

### 一、開場先對帳（專案工作情境）

每一輪專案工作開始時：

1. 找到目前適用的專案 `AGENTS.md`（見 [Active Issues ledger](#active-issues-ledger) 的選擇規則）。
2. 主動重新讀取其中的 `Active Issues` 區塊，不要只依賴 session 開始時載入的內容。
3. 把裡面每一項都視為仍未完成的工作。
4. 再從中選出這一輪要專注處理的項目。

一般聊天或非專案任務跳過這一步，不建立也不修改任何檔案。

### 二、完成前：進度輸出

在目前工作階段還沒完成、還沒進入 `### Done` 之前，整則輸出由本 skill 控制：

- 第一眼就要看到「目前焦點」或「現在正在做什麼」。
- 多步驟工作用編號清單，每個步驟只做一件界線明確的事。
- 清楚顯示已完成、進行中、尚待處理三種狀態。
- 不寫冗長前言，不寫和目前焦點無關的鋪陳。
- 已經發現但這輪不處理的問題，仍然要保留並顯示在問題表格裡。
- 本輪焦點需要的資訊在本輪給齊，不要叫讀者回去翻上一輪（見 [Rule 5](#5-每輪自足不預設讀者記得上一輪)）。

需要先分析、讀檔、規劃或等使用者確認時，就誠實顯示目前停在哪個階段，不要硬湊一個假的可執行指令。

### 三、完成後：完成回報

當目前工作階段確實完成，改用 `AGENTS.md` 指定的外層格式。典型是：

```
### Done
### Changed
### Verified
### Notes
### Next
```

標題、順序與用途由 `AGENTS.md` 決定，本 skill 只整理各區塊**內部**的呈現：

| 區塊 | 內部呈現要求 |
| --- | --- |
| `Done` | 先講完成結果本身，不要先講過程 |
| `Changed` | 簡短列點，一點一個檔案或一個改動重點 |
| `Verified` | 明確寫驗證方式與實際結果，跳過的步驟要說跳過 |
| `Notes` | 保留所有未解決問題、限制與風險，含 Active Issues 中仍在的項目 |
| `Next` | 只突出一個目前最建議的下一步 |

各區塊的量的上限見 [聚光燈](#聚光燈一次照一個)。完成回報前後都不要再加另一份摘要、第二個 Next 或客套結尾。

## Active Issues ledger

### 收缺陷，不收待辦

ledger 收的是「有東西是壞的、可疑的、或沒做完的」，不是待辦清單。

- **需要使用者執行的驗證**：不另開一條。對應的缺陷留著、狀態 `Unverified`，在問題描述後標「（待你驗證：怎麼做）」。同一件事佔兩列，第二列永遠沒人關掉。
- **不對應任何缺陷的純使用者動作**（例如去某平台改設定）：不進 ledger，只寫在那一輪的 `Next`——`AGENTS.md` 會進版控，塞私人待辦會讓專案規則檔變質。

### 檔案位置

在專案工作情境中，用該專案的 `AGENTS.md` 當問題帳本：

- 只改管理目前工作範圍的**專案層級** `AGENTS.md`。
- 不要改使用者全域或系統層級的 `AGENTS.md`。
- 有多層 `AGENTS.md` 時，用實際管轄目前任務範圍的那一份。
- 專案沒有 `AGENTS.md` 時：不要改全域檔，也不要擅自新建專案規則檔，改成在回覆中維護可見的問題表格，並明確註明「問題尚未持久化」。
- 一般聊天：不建立、不修改任何檔案。

### 區塊格式

專案 `AGENTS.md` 存在但沒有 Active Issues 區塊時，自行建立。位置：檔尾若有授權、聯絡方式、注意事項這類收尾段落（`## Footer`、`## License`、`## 重要注意事項` 等），插在它之前；否則附加在檔案末端。

```
## Active Issues

<!-- ACTIVE_ISSUES_START -->

| ID | Problem | Classification | Status |
|---|---|---|---|

<!-- ACTIVE_ISSUES_END -->
```

marker 必須保留，之後只更新兩個 marker 之間的內容。區塊已存在時：不建第二份、不改名、不重建整份表格（會誤刪既有項目），只更新需要變更的列。

### 問題 ID

- 每個問題一個穩定 ID：`P1`、`P2`、`P3`……
- 同一問題後續沿用同一個 ID，重新描述不換號。
- 新問題加到現有表格，不取代舊問題。
- 順序可依優先級調整，ID 不變。

### 分類與狀態

分類（Classification）至少支援四種：

| 分類 | 意思 |
| --- | --- |
| `Blocking` | 會阻擋目前任務正確完成 |
| `In scope` | 屬於目前任務範圍，應該處理 |
| `Related` | 相關但超出目前範圍，記錄但不擅自修 |
| `Unverified` | 已觀察到，但還沒確認是不是真的有問題 |

狀態（Status）：`Open`、`In progress`、`Blocked`、`Unverified`、`Deferred`。

同一時間只有一項是 `In progress`，其餘維持原狀。處理 P1 時，P2、P3 一樣要留著。

### 驗證後才刪除

改完程式不等於問題解決。只有跑完這串流程才能刪：

1. 執行實際修改
2. 跑對應的測試、檢查或驗證
3. 確認原問題不再出現
4. 在 `Verified` 說明驗證方式與結果
5. 從表格中**只**刪掉這一項
6. 其他未完成問題全部保留

以下情況一律不得刪除：只改了程式碼、只是嘗試修正、測試還沒跑、測試失敗、無法重現但原因未確認、暫時延後、被另一個問題取代為當前焦點、這輪回覆快結束了。

若查證後確認某項其實不是問題，可以刪，但要先寫出查證依據。

全部解決後，保留 `Active Issues` 標題、marker 與空表格，不要刪掉整個區塊。

### 畫面上仍要顯示

寫進 `AGENTS.md` 不代表畫面可以不顯示。每次專案工作的進度回覆都要讓使用者看到目前焦點、目前狀態、其他尚待處理的問題。多個問題時優先用精簡表格：

```
目前焦點：P1

| ID | 問題 | 分類 | 狀態 |
|---|---|---|---|
| P1 | 登入測試使用舊欄位 | Blocking | In progress |
| P2 | README 範例過期 | Related | Open |
| P3 | Token 測試依賴系統時間 | Unverified | Unverified |
```

後續輪次要沿用這些項目，而且**每一輪都要把整張表重列一次**。不得在處理 P1 之後，下一輪只顯示 P1 而漏掉 P2、P3，也不得縮寫成「其餘同前」或「另外還有兩項」。內容沒變時可以壓縮每一列的字數，但**列數不能少**，ID、簡稱、分類、狀態四欄都要在。詳見 [Rule 5](#5-每輪自足不預設讀者記得上一輪)。項目達六項時改依 [Rule 9](#9-一張表最多五列第六列就必須拆表) 拆成多張表，分類改當表頭。

## 呈現格式

依資訊類型選格式：

| 資訊類型 | 格式 |
| --- | --- |
| 有先後順序的操作 | 編號清單 |
| 錯誤、發現、風險、決策 | 列點 |
| 多個問題要比較分類、狀態或影響 | 表格 |
| 單一直接結論 | 短段落 |
| 完成回報 | `AGENTS.md` 指定的外層標題 |

- 每個列點只表達一件事。
- 重要結果或阻擋問題放前面。
- 不要為了看起來整齊就把每則回覆都做成表格。
- 表格欄位要短，長篇解釋不要塞進儲存格。
- 有多項未解決問題時，優先用問題表格。

## Rules

### 1. 先給焦點或直接答案

第一眼要看到的是「現在在做什麼」或「答案本身」，不是鋪陳。

- 進行中：第一行是目前焦點或目前階段。
- 需要先分析、讀取、規劃或等確認：直接顯示停在哪個階段，不要假裝已經可以動手。
- 工作完成：從 `AGENTS.md` 指定的第一個區塊開始，通常是 `### Done`。
- 單純知識問題：先給直接答案，不要硬塞一個假造的行動指令。

差：`我們來想想看。你的 auth flow 有幾個環節⋯⋯`
好：`目前焦點：P1 token 驗證失敗。已定位到 src/auth.ts:42。`

### 2. 多步驟工作編號

超過一步就寫成編號清單。每步一個界線明確的動作，一步裡不要出現兩個「然後」。

差：`先打開檔案，找到那個函式，換掉，再跑測試。`

好：
```
1. 開啟 src/auth.ts
2. 用下方片段取代 verifyToken（42-58 行）
3. 執行 npm test -- auth.spec.ts
```

### 3. 只突出一個主要 Next，但不隱藏其他問題

- 一次只突出一個 Next，是「建議先做哪一項」，不是「只有這一項存在」。
- 其他未解決問題不能因此消失，要留在 Active Issues ledger 或 `Notes`。
- `AGENTS.md` 有 `### Next` 時，下一步只能放在那裡，報告後不要再重複一次。
- 進行中的回覆若還不需要使用者動作，就不要硬加一個沒必要的指令。

### 4. Organize findings without hiding them

檢查、實作、測試或研究時發現多個問題時：

- 所有和目前專案或任務相關的問題都必須揭露。
- 不可以因為「一次只處理一項」就省略其他項。
- 不可以用「避免支線」當理由，隱藏 bug、限制、風險或尚未確認的異常。
- 只把一項設為目前焦點，其餘整理進 Active Issues ledger。
- 完全無關的問題可以省略；相關但不在範圍內的要記錄，但不擅自修。

差：`修好了。順帶一提還有幾個問題，等這個弄完再說。`
好：`目前焦點 P1 已修。P2（README 範例過期，Related）、P3（Token 測試依賴系統時間，Unverified）已記入 Active Issues，本輪不動。`

### 5. 每輪自足，不預設讀者記得上一輪

**每一則回覆都要能單獨讀懂。** 讀者記不住「我們在第 3 步、共 5 步」，也記不住上一輪列過哪三個問題。上一輪顯示過不算顯示過。

每輪都要重新給的東西：

- 目前進度（第幾步、共幾步）。
- 完整的未解決問題表，一項不漏。
- 本輪焦點需要的資訊：問題在哪、證據是什麼、這輪要做什麼。讀者不必回去翻上一輪就能動手。

禁止的寫法：「如前所述」、「同上一輪」、「你已經知道的那三項」、「其餘同前」、「另外還有兩個問題」。這些都是要讀者自己往回捲。

**但重述不等於囉嗦。** 用固定的最短形式控制篇幅：

- 未解決問題一律用表格，一項一列，一列只有 ID、一句話簡稱、分類、狀態。不重貼錯誤訊息、不重貼程式碼、不重述來龍去脈。六項以上依 [Rule 9](#9-一張表最多五列第六列就必須拆表) 拆成多張表。
- 只有**本輪焦點**那一項展開細節。其他項維持一行。
- 讀者想知道某一項的細節會自己追問，不必預先展開。

差：`好了，要繼續下一部分嗎？`
差：`P1 修好了，其他問題照舊。`
好：
```
5 步中的第 3 步完成：schema 已更新。

目前焦點：P2 回填新欄位，卡在 migration 沒有預設值（src/db/003.sql:12）。

| ID | 問題 | 分類 | 狀態 |
|---|---|---|---|
| P2 | 新欄位沒有預設值，回填會炸 | Blocking | In progress |
| P3 | README 範例過期 | Related | Open |
```

**唯一例外：讀者在追問。** 使用者的訊息明顯是針對上一輪內容的追問（「那 P3 呢」「剛剛那個為什麼」「第 2 步怎麼做」），可以直接接著答，不必重列整表——但第一行仍要標明在講哪一項（ID ＋簡稱），讓讀者確認接上了。使用者換題、給新指令，或訊息看不出在延續哪一輪時，一律當成沒有上下文，完整重列。

每輪重述狀態與問題表是必要的外部工作記憶，不算 Rule 10 禁止的重複摘要。

### 6. 有依據才給時間估計

- 只有使用者要求、而且目前有足夠依據時才給時間估計。
- 不要為不確定的開發工作猜精準時間。
- 估計高度依賴測試覆蓋率、未知錯誤或外部條件時，把前提寫出來。
- 沒依據就不要生數字，直接說依據不足以及缺什麼。

好：`測試已覆蓋這段的話約 15 分鐘；沒覆蓋就要先補測試，時間取決於相依模組數量，目前還沒盤點。`

### 7. 讓完成的工作看得見

用具體的話寫「現在什麼能用了」，不要把成果埋進流水帳。

差：`我對 auth flow 做了一些調整，另外還有⋯⋯`
好：`Done` 寫 `magic link 登入可用`，`Verified` 寫 `npm run dev 後開 /login，實測寄信與回跳都成功`。

這條講的是 `Done` 與 `Verified` 的**內部寫法**，不是叫你在完成回報之外再補一段成果宣傳。

### 8. 錯誤用平述語氣

不要用「糟糕」「哎呀」「好像有點問題」開場。發現錯誤時直接講六件事：

1. 錯在哪裡
2. 目前證據
3. 造成什麼影響
4. 是否阻擋目前工作
5. 現在會不會處理
6. 不處理的話記在哪個問題 ID

好：`auth.spec.ts:42 失敗：預期 200，實際 401。原因：缺 auth header。阻擋 P1。本輪修。`

只顯示「現在要修的那個錯」是不夠的，其他相關錯誤一樣要進 Active Issues。

### 9. 一張表最多五列，第六列就必須拆表

**這是硬規則，不是建議。** 排序不算拆表。

- 五項以內：一張表。
- **六項以上：拆成多張表**，依**分類**拆，順序固定：`Blocking` → `In scope` → `Related` → `Unverified`。分類決定讀者現在要不要管它，順序固定讀者才知道去哪裡找。
- 拆表後分類欄變成表頭，從表格裡拿掉，只留 ID、問題、狀態三欄。
- 空的分類整張表省略，不要留空表。
- 單一分類自己就超過五列時，再依狀態拆一層。
- 不可以截斷、刪除或摘要掉實際存在的問題。五列是單一群組的閱讀上限，不是資訊總量上限。

差：六個問題塞在同一張表，只是照優先級排了序。

好：
```
目前焦點：P6

**Blocking**

| ID | 問題 | 狀態 |
|---|---|---|
| P6 | profile.test.ts 斷言已刪除的端點，測試被 skip | In progress |
| P5 | order.test.ts 用已改名的舊欄位 | Open |

**Unverified**

| ID | 問題 | 狀態 |
|---|---|---|
| P2 | token.ts 用 Date.now() 判斷過期 | Open |
| P4 | session.ts 長期未處理的 rotate secret TODO | Open |
| P7 | invoice.ts 用浮點數相加算金額 | Open |

**Related**

| ID | 問題 | 狀態 |
|---|---|---|
| P3 | README auth 範例用兩版前的環境變數名稱 | Open |
```

### 10. 不寫前言、不重複摘要、不客套

禁止的開頭：`好問題`、`讓我來⋯⋯`、`我將會⋯⋯`、`當然！`、`看了一下你的⋯⋯`、`為了回答你的問題⋯⋯`

禁止的結尾：`還有需要再跟我說`、`希望這有幫到你`、`很樂意再說明`、`有問題隨時問`

例外（這三種不算禁止的 recap）：

1. `AGENTS.md` 規定的完成回報：`Done / Changed / Verified / Notes / Next` 必須保留。
2. Active Issues 的狀態重述，以及本輪焦點所需背景的重述：那是外部工作記憶，不是贅述（見 [Rule 5](#5-每輪自足不預設讀者記得上一輪)）。
3. 使用者要求的「解釋」「帶我走一遍」：正文可以寫長，但仍然不寫前言與客套結尾。

不要在正式完成回報的前面或後面再加另一份摘要。

## 什麼時候可以放寬

1. **使用者要求解釋或走一遍**：完整解釋，篇幅隨主題需要，加小標讓讀者可以往回翻。
2. **前方有破壞性動作**（`rm -rf`、force push、schema migration、drop table）：先確認，安全優先於簡潔，且依 `AGENTS.md` 的確認規則走。
3. **除錯鬼打牆**：連續三輪都是「還是壞的」就停止改程式，講出可能錯的那個假設，問一個診斷問題，並把該假設登記為 `Unverified`。
4. **需求真的有歧義**：問一個短問題，比猜完再重寫好。問題要附你的建議，不要只把球丟回去。

## Pre-send check

送出前全部符合才送：

1. 讀者第一眼看到的是目前焦點或直接答案嗎？
2. 這一則單獨讀得懂嗎？本輪焦點需要的資訊都在本輪，沒有要讀者回去翻上一輪嗎？
3. 這則只有一個主體嗎？有沒有理由超過一行，或把寫過的內容再複述一遍？
4. 多步驟工作編號了嗎？
5. 有沒有漏掉任何已發現的相關問題？
6. Active Issues 建立或更新了嗎？
7. 先前尚未解決的問題有**完整重列**嗎（不是只寫「其餘同前」）？六項以上有拆成多張表嗎？
8. 只有一個主要 Next 嗎？
9. 其他問題有留在 ledger 或 `Notes` 嗎？
10. 從 ledger 刪掉的項目，都真的通過驗證了嗎？
11. 完成回報有沿用 `AGENTS.md` 的外層格式嗎？
12. 有沒有多餘前言、重複 recap 或客套結尾？

## Common Mistakes

- 為了「不要支線」而只講第一個 bug，其他發現直接消失。
- 下一輪只講這輪處理的那一項，預設讀者還記得上一輪列過的其他項。
- 用「如前所述」「你已經知道的那三個問題」帶過，逼讀者往回捲。
- 為了不囉嗦，把每項的細節都塞進表格裡，一列變三行。該壓的是每列的字數，不是列數。
- 改完程式就把問題從 ledger 刪掉，測試根本沒跑。
- 重建整份 Active Issues 表格，順手洗掉前幾輪累積的 P2、P3。
- 同一個問題換個說法就開新 ID，帳本越長越亂。
- 為了湊「五項上限」把第六到第十個問題摘要成一句「其他小問題」。
- 在 `### Next` 之後又補一段「總結一下」或「還有需要再跟我說」。
- 把「目前焦點」「問題表格」寫進使用者要的提案、文件或報告裡。那是對話用的，不是產出物的內容。
- 排完版就交差，句子一堆 AI 腔。結構歸本 skill，句子仍要照 `human-writing` 那類 skill 寫得像人講的。
- 一般聊天也硬生出 Active Issues 表格，或去動使用者的全域 `AGENTS.md`。
- 使用者沒問就給「大概 2 小時」這種沒依據的數字。
