# 交付前自檢清單與病徵診斷表

## 三分鐘快檢流程

時間不夠跑完整清單時，按順序做這四件事，攔截率最高：

1. **30 秒**：只讀第一段，問「結論和 ask 都在嗎」。不在，先修這裡——這是唯一保證被讀的一段。
2. **60 秒**：把所有標題（或關鍵線主張句）依序連讀，問「連起來是完整論證嗎」。
3. **60 秒**：找 ask，問「對方能當場回答嗎」。「請指示」不能當場回答，「A 或 B、週五前」可以。
4. **30 秒**：全文搜尋三個危險詞——「其他」（分組錯）、「可能有點」（未量化）、「我做了」（activity 混入）。

## 交付前自檢清單

逐項打勾，任一項不過就回去修，不要「大致可以」。

### 結構層

- [ ] **第一段測試**：只讀第一段（email 為主旨＋首句），受眾知道結論與 ask 嗎？
- [ ] **主張測試**：governing thought 句尾加「我不同意」，語意通順嗎？不通順代表它是主題不是主張。
- [ ] **向上測試**：關鍵線各點加總，推得出頂點嗎？有沒有多餘或缺漏？
- [ ] **縱向 Q&A**：每個主張下一層的第一句，直接回答該主張引出的 why / how / so what 嗎？
- [ ] **開場變體**：選用的 SCQA 變體符合受眾狀態（趕時間→ASC、抗拒→演繹鋪路、焦慮→QSCA、自滿→CSA）？

### 分組層

- [ ] **複數名詞測試**：每組能用一個複數名詞概括嗎（三個原因、兩個方案）？
- [ ] **兩兩重疊測試**：任兩組之間，有項目可以同時放進兩組嗎？
- [ ] **無「其他」桶**：全文沒有「其他 / 雜項 / 補充事項」分組？
- [ ] **遞進邏輯**：組內順序打亂後讀起來有差嗎？沒差就重排（預設重要性遞減）。

### 內容層

- [ ] **findings not activities**：逐條掃描，「我做了 X」都已改寫成「發現 Y、所以 Z」或移附錄？
- [ ] **證據著地**：每個主張句下至少一條具體證據？沒有的已標明為假設？
- [ ] **量化**：影響都有數字（錢 / 天 / 人 / %）？「有點」「可能不小」都已消滅？

### 格式層

- [ ] **標題連讀**：所有節標題或投影片標題依序連讀，等於完整論證？
- [ ] **ask 具體**：對方知道要「決定什麼、給什麼、何時之前」？
- [ ] **向上匯報帶選項**：有選項 A/B、你的推薦、推薦理由？
- [ ] **壞消息順序**：若含壞消息，四段順序（結論→影響→已採取行動→需要的支援）未被打亂、未被鋪陳稀釋？
- [ ] **篇幅**：符合 length_limit？超出的部分已進 appendix 而不是硬塞正文？

## 病徵診斷表

讀者反應是最準的檢測器。症狀 → 病因 → 修法：

| # | 症狀 | 病因 | 修法 |
|---|---|---|---|
| 1 | 讀者說「所以呢？」 | governing thought 是主題不是主張 | 改寫成有立場、可被反對的完整句子，並掛上 ask |
| 2 | 讀者說「太長了，重點是什麼」 | 結論埋在中後段（buried lede） | 把結論搬到第一段，背景壓進 SCQA 的 S，細節移 appendix |
| 3 | 讀者讀完不知道要做什麼 | 沒有 ask 或 ask 太模糊 | 行動項置底加粗，寫成「誰、做什麼、何時之前」 |
| 4 | 主管反問「為什麼是這三點？」 | 關鍵線不 MECE，或湊不出複數名詞 | 換分類軸重新分組，跑兩兩重疊＋集體窮盡測試 |
| 5 | 讀者跳著讀就迷路 | 標題是名詞短語不是主張句 | 標題行動化，改到連讀等於完整論證為止 |
| 6 | 對方直接質疑結論本身 | 受眾抗拒卻用了歸納直給 | 改演繹：先找對方會點頭的大前提（他的目標、公司規定、既有共識） |
| 7 | 對方聽到一半不耐煩、打斷 | 受眾趕時間卻用了演繹長鏈或標準式 SCQA | 改 A-S-C 開門見山，理由歸納並列，最強的放第一 |
| 8 | 週報被說是流水帳 | 寫的是 activities 不是 findings | 每條活動改寫成該活動得出的結論，以結論分組（進度 / 風險 / 需支援） |
| 9 | 分組裡出現「其他」 | 分類軸選錯 | 換軸（時間 / 流程 / 對象 / 內外部），或把重要項升級成獨立組、不重要項刪除 |
| 10 | 壞消息匯報後主管更生氣 | 結論被鋪陳包裹、缺「已採取行動」段 | 套壞消息四段：結論不藏 → 影響量化 → 已採取行動 → 需要的支援 |
| 11 | 被批「不要只丟問題給我」 | 向上匯報沒給選項與推薦 | 補選項 A/B 各附一行代價，標明推薦與理由，ask 窄化到可當場回答 |
| 12 | 簡報每頁都被問「這頁想說什麼」 | 一頁多主張，或標題與內容脫節 | 一頁一主張：拆頁，標題改主張句，刪除不服務該主張的圖表 |

使用方式：交付後收到上述任何反應，先對表找病因再改稿，不要憑感覺加更多內容——九成的病要用「刪與重排」治，不是用「補寫」治。

## 常見誤修

對症下錯藥比不修更浪費時間，三個典型誤修：

- **被說「重點不清楚」就加粗更多字** — 病因是結構（結論位置、標題主張化），不是排版。全文超過三處粗體時，粗體已失效。
- **被質疑論點就堆更多數據** — 對方質疑的往往是「為什麼是這三點」（分組問題）或「憑什麼下這個結論」（縱向斷裂），不是數據量。先修結構，再談證據。
- **被說「太長」就整段等比例刪** — 等比例刪會把結論和 ask 一起刪薄。正確做法：結論與 ask 一字不動，證據每點只留最強一條，其餘全進附件。

## 何時破例

規則有邊界，兩種情境明確不適用結論先行：

- **結論會觸發強烈情緒且關係大於效率時**（裁員面談、重大人事），先處理人再處理事，開場用 Q-S-C-A 緩衝而不是 BLUF 直砸。
- **受眾就是要看過程時**（稽核、法遵、學術審查），過程本身是交付物，activity 不是雜訊。此時金字塔用於「附在最前面的摘要頁」，正文保留完整過程。

破例要明說：在 `delivery_notes` 標明「本次未採結論先行，原因是 X」，讓使用者知道這是判斷，不是漏做。
