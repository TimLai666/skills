# 金字塔原理核心：四規則、Governing Thought、縱橫邏輯、MECE

## 四規則操作化

### 規則一：結論先行

任何溝通單位（整份文件、一個章節、一封 email、一頁投影片）的第一句都是該單位的結論。

- 操作法：寫完初稿後，找到「所以」「因此」「綜上」後面那句話，把它搬到最前面。初稿的結論幾乎總是長在尾巴。
- BLUF 版本：第一句 = 結論 + 需要對方做的事。軍方寫作規範的要求是：收件人只讀第一句也能正確行動。
- 測試：把第一句以外全部刪掉，問「對方知道發生什麼、知道要做什麼嗎」。答案是否，第一句不合格。

### 規則二：以上統下

每個上層陳述必須是其下層內容的概括，每個下層內容必須是上層的解釋或證據。

- 向下測試：遮住下層只讀上層，上層必須自足成立（不需要下層才看得懂）。
- 向上測試：只讀下層各點，把它們加總，必須能推出上層那句話，不多也不少。下層加總推不出上層 = 支撐不足；下層有內容跟上層無關 = 該內容移走或刪除。

### 規則三：歸類分組

同一組內的項目必須屬於同一邏輯範疇。

- 複數名詞測試：這一組能否用一個複數名詞概括——三個「原因」、四個「證據」、兩個「方案」、三項「風險」。湊不出複數名詞 = 不是同類，不能放同一組。
- 「原因」和「方案」不能混在同一組；「現況事實」和「未來建議」不能混在同一組。混了就拆層或拆組。

### 規則四：邏輯遞進

組內項目必須有明確排序邏輯，三選一，不可混用：

- 時間順序：步驟、流程（先做 A 再做 B）。用於行動計畫、導入步驟。
- 結構順序：空間或組成（北區/中區/南區；產品/價格/通路）。用於現況盤點、責任分工。
- 重要性順序：程度遞減（最重要的放第一）。用於理由、風險、建議。
- 測試：隨機打亂組內順序，如果讀起來沒差，代表沒有遞進邏輯，重排。商務溝通預設用重要性順序，最強的先講。
- 選錯的代價：把「理由」按時間順序排（想到什麼寫什麼），最強理由常落在中段被略過；把「步驟」按重要性排，執行的人會做錯順序。

## Governing Thought 提煉五步

從一堆素材到金字塔頂點的一句話：

1. **素材倒出** — 把所有發現、數據、事件列成清單，一行一條，不篩選。
2. **分組** — 把相關條目聚類，每組貼一個複數名詞標籤（通過規則三測試）。
3. **每組結論** — 對每組問「這幾條加起來說明了什麼」，寫成一句 so-what 句。禁止寫成標籤（「關於成本的三個發現」不是結論，「成本已超支 15% 且無收斂跡象」才是）。
4. **組間結論** — 對這幾句組結論再問一次「加起來說明了什麼」，得到候選頂點。
5. **頂點改寫** — 用受眾決策語言改寫候選頂點：對方讀完這句要知道「該做什麼決定」。把「情況很嚴重」改成「需要在本週五前決定是否啟用備案」。

### 實作演練：從素材到頂點

素材（週報原料，倒出後）：

- 週二和 A 客戶開季度會，對方採購說預算凍結到 Q4
- B 客戶追加訂單 200 萬
- 新業務入職兩週，已能獨立跑客戶
- C 客戶投訴交期，已補救
- 本月業績達成率 92%
- 競品在 A 客戶端報了低價

分組（貼複數名詞標籤）：

- 「兩個業績訊號」：達成率 92%；B 客戶追加 200 萬
- 「兩個 A 客戶風險」：預算凍結到 Q4；競品低價切入
- 「兩件營運事項」：新業務可獨立作業；C 客戶投訴已補救

每組結論（so-what 句）：

- 業績訊號 → 「本月可達標，B 客戶追加補上了缺口」
- A 客戶風險 → 「A 客戶 Q4 前有流失風險，競品正在卡位」
- 營運事項 → 「團隊產能與客訴皆穩定，無需介入」

組間結論 → 頂點改寫（受眾決策語言）：

> 「本月達標無虞，但 A 客戶流失風險升高——建議下週由您出面與對方高層餐敘卡位，我來安排。」

對照：多數人的週報會把六條素材按時間順序流水帳列出，頂點那句話根本不會出現。

### 主張 vs 主題判別

| | 主題（topic） | 主張（assertion） |
|---|---|---|
| 形式 | 名詞短語 | 完整句子 |
| 立場 | 無 | 有，可以被反對 |
| 例 | 「Q3 供應鏈風險分析」 | 「Q3 供應鏈風險已高到需要立刻啟用第二供應商」 |

測試：在句尾加「，我不同意」。「Q3 供應鏈風險分析，我不同意」語意不通 = 主題；「需要立刻啟用第二供應商，我不同意」語意通順 = 主張。金字塔每個節點（頂點、關鍵線、標題）都必須是主張。

## 縱向問答法

上層陳述會在讀者腦中自動引出疑問，下一層的唯一任務是直接回答那個疑問。

疑問類型對照：

| 上層陳述類型 | 引出的疑問 | 下層必須提供 |
|---|---|---|
| 主張型（「應該做 A」「X 是問題」） | Why — 憑什麼這樣說 | 理由、證據 |
| 行動型（「我們將執行 A」） | How — 具體怎麼做 | 步驟、計畫、分工 |
| 事實型（「數據顯示 X」） | So what — 所以呢 | 含義、影響、建議 |

檢查程序：

1. 對金字塔每個節點，寫下「這句話會讓讀者問什麼」。
2. 檢查下一層的第一句是否直接回答該疑問。
3. 回答的是另一個問題 = 結構斷裂。常見斷裂：上層說「應該做 A」，下層講 A 的背景故事而不是做 A 的理由——背景屬於 SCQA 的 S，不屬於支撐層。

斷裂 before/after：

- 上層主張：「建議改用月結制」（引出的疑問是 why）
- Before（斷裂）：「目前我們採用週結制，這個制度是 2019 年導入的，當時的考量是……」——回答的是「現況是什麼」，不是 why。
- After（接上）：「因為月結可讓對帳工時減半，且八成供應商已採月結；週結是 2019 年為配合舊系統的限制，該限制已不存在。」——第一句直接回答 why，背景降為補充。

節點沒有引出任何疑問（讀者不會想追問）代表這句話是廢話或常識，刪除。

## 橫向邏輯：歸納 vs 演繹

### 歸納法

並列同類論據，共同支撐上層結論。

- 結構：結論 ← 論據 1 + 論據 2 + 論據 3（同類並列，互相獨立）
- 必過複數名詞測試：三個論據要能被「三個原因 / 三項證據 / 三個效益」概括。
- 排序：最強論據放第一。讀者可能只讀第一點。
- 強度：抽掉任何一個論據，結論仍應大致成立。若結論完全掛在單一論據上，那不是歸納，考慮改演繹或補證據。

### 演繹法

大前提 → 小前提 → 結論，環環相扣。

- 例：所有預算超過 500 萬的專案需要 VP 審批（大前提，對方已接受的規則）→ 本專案追加後預算達 620 萬（小前提，事實）→ 因此本案需送 VP 審批（結論，邏輯必然）。
- 大前提的來源：公司規章、對方說過的話、雙方共識、產業常識。大前提若不被對方接受，全鏈崩塌——先確認大前提是對方會點頭的。
- 鏈長限制：不超過三步。演繹鏈越長，讀者越不耐煩，中途斷點越多。

### 選擇規則

| 受眾狀態 | 選擇 | 原因 |
|---|---|---|
| 抗拒你的結論 | 演繹鋪路 | 先讓對方同意大前提，結論成為邏輯必然，反對成本變高 |
| 趕時間 / 高階主管 | 歸納直給 | 結論先出，理由並列，對方讀到滿意就能停 |
| 中立且議題複雜 | 頂層歸納、局部演繹 | 整體結構快，最有爭議的那一組用演繹補強 |
| 判斷不出來 | 預設歸納 | 商務溝通八成場景適用，風險最低 |

注意：同一層只能用一種邏輯。歸納組裡混進一條「因為規定所以必須」的演繹句，會讓複數名詞測試失敗，把它拆出去或改寫成並列論據。

## MECE 實作檢查

- **兩兩重疊測試（ME）**：任取兩組，問「有沒有一個項目可以同時放進這兩組」。有 = 重疊。修法：重新定義兩組的邊界，或換分類軸。
- **集體窮盡測試（CE）**：問「對這個決策重要的東西，有沒有哪個放不進任何一組」。有 = 缺口，補組或擴組。窮盡的邊界是「對本次決策重要的範圍」，不是宇宙窮盡——不要為了形式完整硬加無關組。
- **「其他」桶警訊**：分組出現「其他 / 雜項 / Misc」= 分類軸選錯。修法三選一：換一個切分軸（時間 / 流程階段 / 利害關係人 / 內外部 / 人事物）；把「其他」裡的重要項升級為獨立組；確認剩餘項不重要後直接刪除。
- **3±2 規則**：關鍵線 3 點最穩；2 點可以（通常配演繹）；上限 5 點。超過 5 點 = 分組層級不夠，往上再抽一層（例如 7 個問題先分成「內部 / 外部」兩類）。
- **常用現成軸**：流程（前 / 中 / 後）、對象（客戶 / 內部 / 供應商）、二分（質 / 量、內 / 外、短期 / 長期）、公式拆解（營收 = 客數 × 客單價）。先試現成軸再自創。

## 四規則的取捨優先序

時間不夠把四規則跑完整時，按這個順序保底：結論先行 > 以上統下 > 歸類分組 > 邏輯遞進。第一條做到，溝通就及格；後三條決定的是說服力上限，不是及格線。反過來說，組內排序再漂亮，結論埋在最後一段，一切歸零。
