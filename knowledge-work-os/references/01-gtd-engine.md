# GTD 引擎：五步驟操作化

GTD（Getting Things Done, David Allen）的核心主張：大腦是用來產生想法的，不是用來儲存想法的。系統的價值在於把所有「開放迴圈」移出腦袋，放進一個你信任的外部系統。

理論支撐：Zeigarnik effect 指出未完成事項會持續佔用工作記憶、產生侵入性念頭。Masicampo & Baumeister（2011）的實驗進一步發現：**不需要真的完成任務，只要為它寫下具體計畫，侵入性念頭就會顯著減少，對後續任務的干擾也會消失**。這是 GTD 有效的認知機制——寫下來的不是備忘，是釋放工作記憶的動作。但注意：計畫必須具體（做什麼、何時、何地），模糊的「記得處理」沒有這個效果。

## Step 1: Capture 收集

### 收集點盤點

先列出所有「事情進來的地方」，常見的有：

- Email 信箱（可能不只一個）
- 通訊軟體（Slack / Teams / LINE 工作群）
- 會議中口頭答應的事
- 腦中突然想到的事
- 紙條、便利貼、手機備忘錄
- 別人桌邊攔截的請託
- 專案管理工具的指派（Jira / Asana / Notion）

### 合併原則

收集點越多，漏接率越高，檢查成本越高。目標：**最終收集點 ≤ 3 個**。

- Email 保留（它本身就是收件匣）
- 通訊軟體的待辦：當場轉寄給自己或丟進任務工具，不留在聊天紀錄裡。聊天紀錄不是收件匣，它會沉下去。
- 口頭請託與腦中念頭：統一進一個「快速捕捉」入口（手機備忘錄、任務工具的 inbox、隨身筆記本擇一）
- 紙條便利貼：廢除，改用上面的快速捕捉入口

### 快速捕捉入口的 10 秒標準

從「想到」到「寫完」必須在 10 秒內完成，超過這個摩擦，人在忙碌時就會跳過捕捉、回到用腦記——收集點失守多半從這裡開始。檢驗方式：

- 手機：從鎖定畫面到輸入完成，幾個動作？（widget 或鎖屏捷徑 > 開 app 找 inbox）
- 電腦：有沒有全域快捷鍵可以直接跳出輸入框？
- 會議中：允許降級成「在紙上寫一行」，但會後必須倒進入口

捕捉時只寫得出模糊短語沒關係（「印表機那件事」），釐清是 clarify 的工作，捕捉的唯一 KPI 是不漏。

### 全收集原則

第一次建立系統時，做一次完整的 mind sweep：把所有掛心的事全部倒出來，工作的、私人的、大的、小的、模糊的。標準是「倒完之後，腦中想不到任何還沒寫下的掛心事」。一般人第一次做會倒出 50-150 條，耗時 30-60 分鐘。

收集階段不分類、不判斷、不處理，只倒。分類是下一步的事，混在一起做兩件事都做不好。

## Step 2: Clarify 釐清

對收件匣裡的每一條，依序問：

### 可執行判斷樹

```
這是什麼？需要行動嗎？
├─ 不需要行動
│   ├─ 沒用了 → 丟棄（trash）
│   ├─ 以後可能想做 → someday-maybe 清單（孵化）
│   └─ 以後可能要查 → reference 歸檔
└─ 需要行動
    ├─ 下一步行動 2 分鐘內可完成 → 立刻做（do）
    ├─ 我不是最適合做的人 → 委派（delegate）→ 記入 waiting for
    └─ 需要我做且超過 2 分鐘 → 延後（defer）
        ├─ 一步能完成 → 寫成下一步行動，進 next actions
        └─ 需要多於一步 → 這是專案，進 projects 清單
            並為它定義出「第一個下一步行動」進 next actions
```

### 2 分鐘規則

2 分鐘內能完成的事，追蹤它的成本高於做掉它的成本，所以當場做。注意兩個誤用：

- 釐清時段裡連續做十件 2 分鐘的事沒問題；但在深度工作時段中，2 分鐘規則不適用——那時任何切換都要付 attention residue 的代價（見 02）。
- 「2 分鐘」是實際完成，不是「開個頭」。開頭做一半更糟。

### 下一步行動語法

格式：**動詞開頭 + 可直接執行 + 情境標註**

「可直接執行」的檢驗：想像你只有 10 分鐘空檔看到這條，能不能不經任何思考直接動手？需要先想「所以我到底要幹嘛」的就不合格。

| 不合格寫法 | 問題 | 合格寫法 |
|-----------|------|----------|
| 媽媽生日 | 沒有動詞，不知道要幹嘛 | 打給大姐討論媽媽生日聚餐地點 @電話 |
| 報告 | 是專案不是行動 | 打開 Q3 報告檔案，列出大綱三個主節 @電腦 |
| 研究一下新工具 | 「研究」無法直接動手 | 看完 Notion 官方 10 分鐘入門影片 @電腦 |
| 跟 IT 處理權限問題 | 「處理」不具體 | 寄信給 IT 附上錯誤截圖，要求開通報表系統權限 @email |
| 健身 | 沒有可執行單位 | 週三下班去健身房做 30 分鐘腿部訓練 @外出 |

情境標註（@電話、@電腦、@辦公室、@外出、@等碎片時間）讓你在特定場景下能快速篩出「現在做得了的事」。情境數量 3-6 個就好，太細反而增加維護成本。

## Step 3: Organize 組織

釐清後的東西各有去處，清單體系如下：

### next actions（下一步行動清單）

- 只放「一步可完成、已釐清、輪到我做」的行動
- 按情境分組（@電腦 / @電話 / @外出...）
- 這是你每天實際看著做事的清單

### waiting for（等待清單）

- 所有委派出去、或等別人回覆的事
- **每條必記三件事：等誰、等什麼、從哪天開始等**
- 每週回顧時逐條巡，超過預期日的主動追。沒有這個清單，委派 = 失蹤

格式範例：

```
等 Amy 回傳設計稿 v2（6/28 委派，預期 7/3）
等 IT 開通報表系統權限（7/1 提單 #4482，預期 7/4）
等客戶確認合約條款（7/2 寄出，預期 7/8，超期就打電話）
```

### projects（專案清單）

- 定義：**任何需要多於一步才能完成的成果**，都是專案。「換辦公室印表機碳粉」如果要先查型號再下單，它就是專案
- 專案清單記的是「要達成的成果」，用成果語法：完成時你能明確打勾的狀態描述
- **專案清單與行動清單分離**：專案本身不可執行，可執行的永遠是它的下一步行動。混在同一張清單會讓你看著「Q3 報告」發呆
- 鐵律：**每個活著的專案，任何時刻都必須在 next actions 或 waiting for 裡有至少一條對應項目**。沒有的專案就是停滯專案，每週回顧的首要工作就是抓出它們

專案命名的成果語法對照：

| 不合格寫法 | 問題 | 合格寫法 |
|-----------|------|----------|
| 做 Q3 報告 | 「做」沒有完成狀態 | Q3 報告定稿送出給總經理 |
| 網站的事 | 不知道要達成什麼 | 官網改版上線並通知既有客戶 |
| 招人 | 沒有可打勾的終點 | 資料分析師到職且完成第一週 onboarding |
| 持續改善流程 | 永遠不會完成，不是專案 | 拆成具體成果，或移到「責任範圍」備忘，不進專案清單 |

### someday-maybe（將來也許）

- 不是現在承諾要做、但不想忘掉的事：想學的技能、想做的 side project、別人推薦的書
- 作用是隔離：讓 next actions 保持「全部都是真承諾」的可信度
- 每週回顧掃一眼，偶爾有東西升格為專案，也允許整條刪掉

### reference（參考資料）

- 不需行動、只需要找得到：文件、連結、筆記
- 唯一要求是可搜尋。不要花時間建精美分類，搜尋比分類便宜

### 行事曆

- 只放硬承諾：指定時間的會議、死線、預約
- 不放「希望今天做」的彈性待辦——那是 next actions 的事。行事曆上的東西應該 100% 會發生，一旦混入願望，你就會開始習慣性無視行事曆

## Step 4: Reflect 回顧

系統不會自己保持乾淨。兩個節奏：

### 每日維護儀式（10-15 分鐘，固定在下班前或睡前）

```
1. 清空快速捕捉入口：每條過 clarify 判斷樹，歸位（5 分鐘）
2. 看行事曆：明日硬承諾有哪些、第一場會幾點（1 分鐘）
3. 掃 next actions：挑出明日候選，圈出明日 big 3（3 分鐘）
4. 排明日時間箱（做法見 02，3-5 分鐘）
```

放下班前而不是隔天早上的理由：帶著「明天已經排好」的狀態下班，晚上不被工作念頭糾纏（Zeigarnik 機制）；而且早上的高峰精力該拿來做深度工作，不是排程。

### 每週回顧（30-60 分鐘）

完整腳本在 [03-review-and-load.md](./03-review-and-load.md)。每週回顧是整個系統的心跳。跳過兩週，清單開始腐敗；跳過四週，系統死亡（見下方崩壞模式）。

## Step 5: Engage 執行

當下要做哪一件事？依序用四個準則過濾：

1. **情境**：我現在人在哪、有什麼工具？→ 篩出做得了的
2. **時間**：下個硬承諾之前有多少分鐘？→ 15 分鐘空檔不要開需要 2 小時的深度任務
3. **精力**：現在是高峰還是低谷？→ 低谷做 shallow（回信、整理、行政），高峰留給深度工作（見 02）
4. **優先**：剩下的選項裡，哪件對本週 big 3 貢獻最大？

注意順序：優先級是最後一個過濾器，不是第一個。先問優先級會讓你在錯誤的情境、時間、精力狀態下硬啃重要任務，效率極差還打擊信心。

實例：下午 15:40，16:00 有會，剛開完一場冗長會議精力低迷。

1. 情境 @電腦 → 篩出 12 條做得了的
2. 時間 20 分鐘 → 剔除「寫報告第二節」這類需要完整區塊的，剩 5 條
3. 精力低 → 剔除需要思考的，剩 3 條 shallow（回覆排程信、整理會議記錄、填報銷單）
4. 優先 → 會議記錄跟本週 big 3 有關，做它

錯誤做法是在 15:40 打開報告硬寫 20 分鐘：產出近零，還帶著挫折感進下一場會。

## 系統崩壞模式與重啟

### 常見崩壞模式

| 崩壞模式 | 症狀 | 根因 |
|----------|------|------|
| 清單腐敗 | next actions 上躺著三週沒動的項目，看到清單就煩 | 願望混進承諾；或項目其實是專案沒拆 |
| 收集點失守 | 又開始用便利貼、聊天紀錄、大腦記事 | 快速捕捉入口不夠順手（超過 10 秒就會被繞過） |
| 回顧斷線 | 連續跳過每週回顧 | 回顧被排在低精力時段、或一開始就把回顧做成 90 分鐘大工程 |
| 殭屍專案 | 專案清單一堆，一半沒有下一步行動 | 只在建立時定義行動，之後沒人巡 |
| 工具流浪 | 每兩個月換一個新工具重建系統 | 把系統失效歸因於工具；換工具是最昂貴的拖延 |

### 重啟步驟（系統已死時）

1. **宣告破產**：舊清單全部封存（不刪，移到 archive），不要試圖逐條清理三個月前的殘骸——清理成本高於重建
2. **重新 mind sweep**：以現在為準，重倒一次所有掛心事（30-60 分鐘）
3. **只重建四張清單**：next actions / waiting for / projects / someday，reference 之後再說
4. **立刻排下一次每週回顧**：放進行事曆，選一週裡精力尚可、干擾最少的固定時段
5. **降低標準運轉兩週**：目標是「收集不漏 + 每日 10 分鐘維護」，不是完美系統。兩週存活後再優化

重啟不可恥。多數長期 GTD 使用者都重啟過數次，系統的可信度來自「現在是乾淨的」，不是「從來沒亂過」。
