# 統計門檻：樣本數、工期與執行健檢

本篇的公式都是雙尾檢定、兩獨立組、正態近似。精確計算交給 `scripts/sample_size.py`，這裡的數字用來心算量級與 sanity check。

## 樣本數速算公式

符號：α = 顯著水準、power = 1 − β、δ = 絕對 MDE（兩組差值）、σ = 指標標準差。

**連續型指標**（AOV、停留秒數、每人營收），α = 0.05 雙尾、power = 0.8：

```
每組 n ≈ 16 σ² / δ²
```

（精確常數是 2 × (1.96 + 0.8416)² ≈ 15.7，用 16 方便心算，略保守。）

**比例型指標**（轉換率、點擊率），一般形式：

```
每組 n = (z_{α/2} + z_β)² × (p₁(1−p₁) + p₂(1−p₂)) / (p₂ − p₁)²
```

心算版（p̄ = 兩組平均比例）：

```
每組 n ≈ 16 × p̄(1−p̄) / δ²
```

相對 MDE 與絕對 MDE：「相對 10%」在基準 2% 時 δ = 0.002。行銷場景建議溝通用相對 MDE（比較直覺），計算時換算成絕對值。

### 參數從哪裡拿

- **baseline**：取最近 2–4 個完整週、同受眾同觸發條件的實際數據，不要用年平均（季節性會偏）。
- **連續型 σ**：取與實驗指標同定義、同時間窗的歷史資料算標準差。營收類指標長尾很重，σ 常比直覺大很多；考慮 winsorize（例如 99.5 百分位截尾）後再估，實驗分析時用同一套截尾規則。
- **MDE**：不是「你猜會提升多少」，而是「最小值得做的效果」——低於它，就算效果真的存在，也付不起上線後的維護與複雜度成本。

### 分流單位與分析單位不一致

user 級分流、session 級指標（例如每 session 轉換率）時，同一 user 的多個 session 彼此相關，直接套公式會低估變異數，p 值過度樂觀。對策：指標盡量定義成 user 級（每 user 是否轉換）；必須用 session 級指標時，用 delta method 或 bootstrap 修正變異數（成熟實驗平台多半內建）。

### Triggered Analysis 與稀釋

只有 30% 使用者會走到被改動的頁面時，把全站使用者都算進實驗會稀釋效果：對觸發者的真實效果 10%，全體量測只剩 3%，而樣本需求隨（效果）² 反比暴增約 10 倍。對策：只把「實際觸發改動」的使用者納入分析——但兩組的觸發判定邏輯必須完全一致，否則會自己製造 SRM；報告時再把效果換算回全站影響。

### 變異數縮減（流量不夠時的合法手段）

- **CUPED**：用實驗前同指標的資料當共變量修正，常見可省 30–50% 樣本。平台有支援才用，手刻容易出錯。
- **指標截尾**：winsorize 壓掉極端大單對 σ 的影響（規則事前定）。
- **Triggered analysis**：縮小到真正受影響的人（見上），比放大 MDE 更不犧牲判讀。

## 速查表：基準轉換率 × 相對 MDE → 每組所需樣本

α = 0.05 雙尾、power = 0.8、兩組。所有數字為近似量級；不同計算器因 pooled / unpooled 公式差異會有 ±5% 上下，量級對就好。

| 基準轉換率 | 相對 MDE 5% | 相對 MDE 10% | 相對 MDE 20% |
|---|---|---|---|
| 1%  | ~64 萬 | ~16.3 萬 | ~4.3 萬 |
| 2%  | ~32 萬 | ~8.1 萬  | ~2.1 萬 |
| 5%  | ~12.2 萬 | ~3.1 萬 | ~8,200 |
| 10% | ~5.8 萬 | ~1.5 萬  | ~3,800 |
| 20% | ~2.6 萬 | ~6,500   | ~1,700 |

讀表要點：

- 任務中常引用的「2% 基準、相對 10% → 約 78k/組」與本表 8.1 萬同量級（78k–82k 依公式版本浮動）。
- MDE 減半，樣本數約變 4 倍（δ 在分母平方）。這是流量不足時「放大 MDE」有效的原因。
- 基準率越低，同樣的相對 MDE 越貴。低轉換率業務（B2B 表單、高單價）常直接卡死在這張表上——這就是 Data Sufficiency Gate 要先跑的原因。

## alpha / power 慣例與何時調整

預設：α = 0.05（雙尾）、power = 0.8。這是慣例不是物理定律，以下情況調整：

- **改動不可逆或成本高**（定價、大改版）：α 降到 0.01，寧可多花樣本也不要假陽性。
- **漏掉真效果的代價高**（策略級賭注，錯過就沒了）：power 提到 0.9（樣本約增 34%）。
- **大量低風險探索型實驗**：可接受 α = 0.1，但必須登記並理解假陽性率隨之上升。
- **單尾 vs 雙尾**：預設雙尾。改動理論上也可能傷害指標（幾乎總是如此），用單尾等於偷 alpha。要用單尾必須事前聲明並給理由。

## 工期規則

1. 基本公式：`工期（天）= ceil(每組樣本 × 組數 / 日進實驗流量)`。
2. 向上取整到**完整週**（7 的倍數）：週間與週末的使用者組成、意圖差異很大，只跑週一到週五會偏。
3. 即使樣本一天就夠，**至少跑 1 週，建議 2 週**：吸收週循環與短期波動。
4. 避開或跑滿異常期間：大促、假期、行銷檔期。要嘛整個避開，要嘛完整涵蓋，不要切一半。
5. 上限約 8 週：cookie 汰換、跨裝置汙染、組織機會成本都會侵蝕更長的實驗。
6. 購買週期長的產品（B2B、家電、旅遊）：工期至少涵蓋一個完整決策週期，否則只量到週期前段的行為。

## Peeking 問題：為什麼「每天看、顯著就停」會爆 false positive

機制：α = 0.05 的保證是「只檢定一次」的條件下成立。每天對累積資料重新檢定、一顯著就停，等於給雜訊很多次「碰巧越線」的機會，而且一越線你就鎖定它——p 值會隨機遊走，遲早有一段時間掉到 0.05 以下。

實際 alpha 膨脹量級（兩組其實無差異、每次都用 α = 0.05 檢定、顯著即停；近似值，出自 sequential testing 文獻的經典計算）：

| 期中檢定次數 | 實際 false positive 率 |
|---|---|
| 1（不 peeking） | 5% |
| 2 | ~8% |
| 3 | ~11% |
| 5 | ~14% |
| 10 | ~19% |
| 20 | ~25% |
| 每天看、連看兩個月 | 30% 以上 |

用 A/A 模擬同樣可以重現：模擬一萬次無差異實驗、每日檢定 30 天、顯著即停，約四分之一到三成的實驗會錯誤宣稱顯著。

對策（三選一，事前決定）：

1. **固定樣本設計**（最簡單）：事前算好樣本，跑滿才看 primary 顯著性。期中只允許看兩件事——SRM 與 guardrail 是否崩壞（崩壞可停損，這不算 peeking，因為停損不是宣稱勝利）。
2. **Sequential 設計**：group sequential（O'Brien-Fleming 邊界，預定 2–3 個檢查點、每點用更嚴門檻）或 mSPRT / always-valid p-value（Optimizely 等平台的 Stats Engine 內建）。代價：同一個 MDE 需要比固定樣本多一些的樣本。
3. **Bayesian 設計**：明示先驗，報告後驗機率與預期損失（expected loss），用預先定義的決策規則（如 expected loss < 閾值）停止。注意：Bayesian 不是免死金牌，先驗要寫出來讓人挑戰，決策規則一樣要事前定。

## SRM 檢查（Sample Ratio Mismatch）

定義：實際分到各組的樣本比例偏離設計比例。SRM 代表分流或記錄壞了，此時任何指標比較都不可信。

檢定：對各組樣本數做 chi-square goodness-of-fit。**警報門檻 p < 0.001**（不是 0.05——SRM 檢定每天做、每個實驗做，門檻要低才不會誤警不斷；而真正的 SRM 通常 p 小到 10⁻⁸ 等級，非常好認）。

範例（50/50 設計、共 20,000 樣本）：

- 觀察到 9,900 / 10,100：chi-square = 2.0，p ≈ 0.16 → 正常波動。
- 觀察到 9,700 / 10,300：chi-square = 18.0，p ≈ 0.00002 → SRM 警報，資料作廢。

常見原因（照這個順序查）：

1. Redirect 掉量：variant 需要跳轉，跳轉慢或失敗 → variant 組流失。
2. Bot 過濾不對稱：某組的頁面結構讓 bot 偵測行為不同。
3. 快取 / CDN：某組被快取層吃掉追蹤事件。
4. 分流條件與觸發（trigger）條件不一致：只分析「有觸發改動的人」但兩組觸發邏輯不同。
5. Ramp-up 資料混入：分流比例中途從 10/90 調成 50/50，資料跨期合併。
6. 使用者刪 cookie / 跨裝置，兩組受影響程度不同。

處理原則：找到根因 → 修 → **整批資料作廢重跑**。不可用「事後加權」救回同一批資料。

## AA Test

兩組給完全相同的體驗，用途：

- 驗證分流均勻（會不會有 SRM）。
- 驗證 instrumentation：兩組指標應該無差異；α = 0.05 下約 5% 的指標會「顯著」，明顯超過就是變異數估計或記錄有問題。
- 建立指標的自然波動基準，供之後判讀參考。

必跑時機：實驗平台第一次啟用、分流邏輯改版後、或連續多個實驗出現詭異結果時。

## Novelty / Primacy Effect 辨識

- **Novelty effect**：新東西因為「新」被多點多用，效果隨時間衰減。常見於顯眼的 UI 改動。
- **Primacy effect**：老使用者習慣舊介面，初期效果差，隨學習曲線上升。常見於工作流程改動。

辨識方法：按**首次曝光 cohort** 分組（第 1 天首次看到的人、第 2 天首次看到的人……），看 treatment effect 隨「曝光後天數」的趨勢。效果隨曝光天數衰減 → novelty；上升 → primacy。

注意：直接看 calendar day 的每日效果趨勢會把「新舊使用者混合比例變化」和「效果衰減」混在一起，必須用曝光 cohort 切。

對策：延長工期到效果曲線走平、只用第 2 週之後的資料估長期效果、或對新使用者單獨分析。

## 多重比較

問題量級：20 個指標各用 α = 0.05 檢定，即使全部無效果，期望值上也會有 1 個「顯著」。多變體同理。

處理方式（依情境選）：

- **層級化判讀（建議預設）**：primary 用全 α 下結論；secondary 與 segment 只做支持性解讀、不單獨宣稱顯著；guardrail 用非劣性思維。這樣多數實驗不需要正式校正。
- **Bonferroni**：門檻改為 α/m。簡單、保守，適合指標少（m ≤ 5）且每個都要下結論的場合。
- **Benjamini-Hochberg（FDR）**：控制錯誤發現率，適合大量 secondary 指標的探索式掃描。
- **多變體（A/B/n）**：對「variant vs control」的比較數做 Bonferroni（或 Dunnett 檢定）。變體每多一個，流量就被稀釋一份、比較數又變多——先用 ICE / PXL 砍到 2–3 個變體再上線。
