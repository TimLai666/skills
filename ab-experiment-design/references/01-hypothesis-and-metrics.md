# 假設卡與指標設計

## 假設卡格式

每個實驗開跑前，先寫成一句完整的假設卡：

> 因為觀察到 **[證據]**，我們相信 **[對誰做什麼改動]** 會導致 **[行為改變]**，可由 **[指標]** 的 **[預期變化量]** 驗證。

範例（合格）：

> 因為觀察到 **結帳頁的 session recording 中 38% 使用者在運費欄位停留超過 10 秒後離開**，我們相信 **對購物車金額未達免運門檻的使用者，在結帳頁頂部顯示「再買 $X 免運」提示** 會導致 **更多人回頭加購而非放棄結帳**，可由 **結帳完成率** 的 **相對提升 5% 以上** 驗證。

範例（不合格）：

> 我們覺得新版按鈕比較好看，改了應該會提升轉換率。
> （沒有證據、沒有機制、沒有預期變化量、「轉換率」定義不明）

## 各欄位品質標準

| 欄位 | 合格標準 | 常見不合格 |
|---|---|---|
| 證據 | 來自數據、用戶研究、客訴、heatmap、session recording、過去實驗，且可指出出處 | 「我覺得」「老闆說」「競品有做」（競品有做只是線索，不是證據） |
| 對誰做什麼改動 | 明確受眾 + 單一可描述的改動 | 一次改五個東西綁成一包，贏了輸了都無法歸因 |
| 行為改變 | 說出「使用者行為為什麼會變」的機制，是行為層描述 | 直接跳到「指標會上升」，中間機制空白 |
| 指標 | 對應到唯一的 primary 指標，有明確定義（分子、分母、時間窗） | 「轉換率」但沒說是誰的、哪一步的轉換 |
| 預期變化量 | 具體數字（建議用相對 %），並說明依據：過去同區塊實驗、業界基準、或「最小值得做的效果」 | 「有提升就好」——這會讓樣本數無法計算 |

證據強度分級（寫進假設卡時標註）：

1. 量化行為數據（自家 funnel、實驗歷史）— 最強
2. 質化研究（訪談、usability test、session recording）
3. 間接訊號（客訴、業務回饋、heatmap）
4. 業界慣例 / 競品觀察 — 最弱，只能當起點，不能單獨成卡

## 假設排序：ICE

候選假設超過一個時先排序，不要照提出順序做。

ICE 每項 1–10 分，總分 = 三項平均：

| 維度 | 問題 | 評分錨點 |
|---|---|---|
| Impact | 如果成功，對 primary 指標影響多大 | 10 = 改核心漏斗主路徑；5 = 改次要路徑；1 = 邊角文案 |
| Confidence | 有多少證據支持會成功 | 10 = 自家數據 + 質化研究都指向同方向；5 = 只有單一間接訊號；1 = 純直覺 |
| Ease | 實作 + 上實驗有多容易 | 10 = 文案改動當天上；5 = 一週內；1 = 需跨團隊排程 |

ICE 的弱點：三項都主觀，Confidence 特別容易被提案人灌水。團隊超過一人、或假設之間要搶流量時，改用 PXL。

### ICE 與 PXL 怎麼選

- 一人團隊、候選假設少於 5 個：ICE 夠用，快。
- 多人提案、假設要搶同一批流量、或優先序常吵不完：用 PXL，把爭論變成可稽核的配分。
- 兩者都只是排序工具，不取代假設卡品質標準——不合格的假設卡先退件補寫，不進排序。

## 假設排序：PXL

PXL（CXL / Peep Laja 提出）把主觀評分改成一組客觀化的是非題，有出處才給分：

| 問題 | 配分 |
|---|---|
| 改動位於首屏（above the fold）？ | 是 = 1 |
| 使用者 5 秒內注意得到差異？ | 是 = 2 |
| 是新增或移除元素（而非只調樣式）？ | 是 = 2 |
| 針對高流量或高意圖頁面？ | 是 = 1 |
| 是否解決質化研究（訪談 / 問卷 / usability test）發現的問題？ | 是 = 1 |
| 是否有自家量化數據（analytics / funnel）支持？ | 是 = 1 |
| 是否有 session recording / heatmap 佐證？ | 是 = 1 |
| 實作工時 | < 1 天 = 3；< 1 週 = 2；< 2 週 = 1；更久 = 0 |

規則：

- 證據類題目「說得出出處才給分」，口頭宣稱一律 0 分。
- 配分可依團隊情境調整，但要事前定案，不可為了讓某個假設排前面而改權重。
- 高分假設先做；低分但老闆很想做的，把分數攤開來討論，而不是直接插隊。

## 指標層級

### Primary（唯一）

- 只能一個。它是實驗成敗的唯一裁判，樣本數照它算。
- 選「離改動最近的行為指標」：改結帳頁 → 結帳完成率，不是全站營收（太遠、太雜訊）。
- 必須是這個改動能合理影響的指標，且靈敏度足夠（事件量夠大、變異數不爆炸）。
- 定義寫死：分子、分母、歸因時間窗、去重規則。

Primary 挑選 checklist（全部要過才定案）：

- [ ] 改動到指標的因果路徑，一句話說得完
- [ ] 事件量夠：實驗期間的預期樣本過得了 references/02 的速查表
- [ ] 不是 vanity metric（瀏覽量、粉絲數這類與價值脫鉤的數字）
- [ ] 實驗期內可收斂量測，不用等 90 天後的行為
- [ ] 團隊對定義沒有歧義，不同人算會得到同一個數字

指標定義模板（primary 與 guardrail 都用這個格式寫死）：

```
指標名稱：結帳完成率
分子　　：完成付款事件的不重複 user 數
分母　　：進入結帳頁的不重複 user 數
時間窗　：首次曝光後 7 天內
去重　　：user 級，一人多次結帳只計一次
排除　　：內部員工、bot、測試訂單
```

### Secondary（解讀機制用）

- 漏斗中間步驟、互動深度等，用來解釋「為什麼贏 / 為什麼輸」。
- 不參與 ship / no-ship 判定，只做支持性解讀。

### Guardrail（必設，護底線）

每個實驗計畫書至少過一輪這份清單，不適用的明確寫「不適用 + 理由」：

| Guardrail | 監控方向 | 越線門檻（範例，依 risk_tolerance 調整） |
|---|---|---|
| 營收 / AOV / 每訪客營收 | 不得顯著下降 | 相對下降 > 1–2% 且方向穩定 |
| 退訂 / 退貨 / 退款率 | 不得上升 | 相對上升 > 5% |
| 頁面速度（載入延遲） | 不得變慢 | p75 延遲增加 > 100ms |
| 錯誤率 / crash | 不得上升 | 高於基準帶寬即警報 |
| 客訴 / 支援工單量 | 不得上升 | 相對上升 > 10% 或出現新類型客訴 |
| （訂閱制）取消率、（廣告）品質分數 | 依業務性質 | 事前與使用者議定 |

Guardrail 判讀用「非劣性」思維：不是要它顯著變好，是要能排除「惡化超過門檻」。

## Proxy Metric 陷阱

經典案例：把 email 主旨改得聳動 → 開信率、點擊率大幅上升 → 一個月後退訂率上升、營收下降。點擊是營收的 proxy，但改動同時破壞了 proxy 與 north star 的相關性。

規則：

- proxy 只有在「歷史上與 north star 正相關，且改動不會破壞這個相關」時才能當 primary。
- 用 proxy 當 primary 時，guardrail 必須包含 north star（營收或留存），即使它 underpowered——方向惡化就要警覺。
- 「點擊率上升但下游全跌」的實驗，判定為失敗，不是成功。

## OEC（Overall Evaluation Criterion）

Kohavi 提出的概念：把長期價值濃縮成一個可短期量測的組合指標（例如加權組合轉換、留存、每次造訪時間），讓所有實驗用同一把尺。

注意事項：

- 權重必須事前定案且全組織一致，不可每個實驗自訂。
- 組合指標可能被 gaming：某一項暴漲掩蓋另一項惡化，仍要保留 guardrail。
- 建議：中小團隊先用「primary + guardrail 分層判讀」就好；OEC 留給實驗文化成熟、每季跑數十個實驗的組織，否則權重爭議會吃掉所有會議時間。
