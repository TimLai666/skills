# 判讀決策樹與陷阱

## 進入判讀前的門檻

以下任一項沒過，直接停在「資料不可信」，不進決策樹：

1. SRM 檢查通過（chi-square p ≥ 0.001）。
2. 跑滿事前宣告的樣本數與工期（至少完整週）。
3. 期間沒有發生未登記的改版、檔期異常或追蹤中斷。

## 判讀決策樹

```
primary 結果？
│
├─ 顯著正向
│   ├─ 1. Twyman check：效果好到不像真的（如相對提升 >25%）？
│   │      → 先查 instrumentation、重算一次，查完才往下走
│   ├─ 2. guardrail 全數通過？
│   │      ├─ 否 → 不上線。量化 trade-off（賺多少 vs 賠多少）呈報，
│   │      │       讓使用者做業務決策；可嘗試修改動再重測
│   │      └─ 是 ↓
│   ├─ 3. 事前聲明的主要 segment 方向大致一致？
│   │      ├─ 大 segment 明顯反向 → 查 Simpson's paradox 與分流；
│   │      │       segment 層的結論一律另開實驗驗證
│   │      └─ 一致 ↓
│   └─ 4. 實務顯著性：效果量 × 觸及規模 ≥ 維護成本 + 複雜度成本？
│          ├─ 否 → 統計上贏但不值得上，登記後關閉
│          └─ 是 → 上線 + 登記；重大改動可留 5–10% holdback
│                  觀察長期效果與 novelty 衰減
│
├─ 不顯著
│   ├─ power 夠嗎？（事前算的樣本有跑滿？）
│   │      ├─ 夠 → 接受「在此 MDE 下無可偵測效果」，登記，資源轉向。
│   │      │       看信賴區間：若 CI 已排除「值得做的效果量」，
│   │      │       可放心關閉；若 CI 還包含它，註明「效果可能存在但小」
│   │      └─ 不夠 → 標記 underpowered / inconclusive。
│   │              不可宣稱「證明無效」。要嘛補樣本重跑，要嘛放大 MDE
│   └─ 禁止：不顯著就開始翻 segment 找「其實有效的族群」（見 HARKing）
│
└─ 顯著負向
    ├─ 不上線（這步不用討論）
    ├─ 回到假設卡定位哪一環錯了：證據誤讀？機制假設錯？執行品質差
    │  （設計醜、文案硬、載入變慢）？——執行差 ≠ 方向錯
    ├─ 若與強證據嚴重矛盾（質化研究都說好，數據卻大跌）→ 先查
    │  instrumentation 與執行品質，再否定假設
    └─ 負向結果照樣登記：它阻止了一次傷害上線，價值不亞於勝利
```

## Twyman's Law

> 任何看起來特別有趣或特別漂亮的數字，多半是錯的。

線上實驗的版本：多數實驗的真實效果落在個位數百分比的相對變化；看到相對提升 25% 以上，先當 bug 處理，查完才准慶祝。檢查順序：

1. 事件重複計數（同一轉換記兩次）。
2. 分子分母的單位錯置（session 當 user、含 bot）。
3. 兩組的追蹤程式版本或觸發時機不一致。
4. 時區、日期邊界切齊問題。
5. filter 或排除規則在實驗中途改過。
6. 只統計到部分漏斗（variant 的某頁事件掉了）。

「查了都沒問題」的巨大效果當然存在（例如修好一個壞掉的流程），但那應該事前就預期得到。

另一個相關偏誤：**winner's curse**。underpowered 實驗剛好過顯著線的效果量，系統性高估真實效果（只有抽到大的那次才過線）。上線後實際效益低於實驗數字是常態，對外承諾效益時打個折。

## 邊緣結果的處理

- p = 0.06 不叫「接近顯著」，就是不顯著。判讀紀律建立在事前定好的 α 上，不因結果微調說法。
- 「再多跑一週看看」是 peeking 的變形。固定樣本設計下不允許；要延長只能依「事前寫好的延長規則」（例如 group sequential 的預定檢查點）。
- 同一假設反覆重測直到顯著，等同多重比較。登記時必須記錄同假設的歷史測試次數，判讀時一併揭露。

## 判讀報告用語紀律

| 情境 | 可以寫 | 不可以寫 |
|---|---|---|
| 顯著正向且 guardrail 過 | 「α = 0.05 下顯著，效果 +x%（95% CI …）」 | 「證明有效」「絕對提升」 |
| underpowered 不顯著 | 「inconclusive，power 不足以偵測此 MDE」 | 「證實無效」「兩版沒差」 |
| 事後 segment 顯著 | 「觀察到⋯值得開新實驗驗證」 | 「對某族群有效」 |
| 效果大得異常 | 「待 instrumentation 覆核後更新」 | 直接寫進成果簡報 |

## Holdback：長期效果驗證

重大改版全量上線時，保留 5–10% 使用者維持舊版 4–12 週，用來：驗證效果是否隨 novelty 衰減、量測實驗期看不到的長期指標（留存、回購）。小改動不必做——holdback 本身有維護成本與體驗不一致的代價。

## Simpson's Paradox

整體與分層的結論相反。具體數字範例（分流在裝置維度失衡時）：

| | 桌機 | 手機 | 整體 |
|---|---|---|---|
| A 組 | 1,000 人 / 100 轉換（10%） | 4,000 人 / 80 轉換（2%） | 5,000 人 / 3.6% |
| B 組 | 4,000 人 / 360 轉換（9%） | 1,000 人 / 15 轉換（1.5%） | 5,000 人 / 7.5% |

B 整體 7.5% 大勝 A 的 3.6%，但 B 在**每一個**裝置分層都輸（9% < 10%、1.5% < 2%）。差異全部來自 B 拿到過多高轉換的桌機流量——這其實是分流壞掉，不是 B 比較好。

實務上最常出現的形態：

- **Ramp-up 跨期合併**：第一週 10/90 分流、第二週 50/50，兩期直接相加。規則：不同分流比例期間的資料不可直接合併，要嘛丟棄 ramp 期、要嘛按期分別分析。
- **改動本身改變了流量組成**：variant 在手機上載入變快 → 手機占比上升 → 整體均值被組成變化拉動。
- 防線：SRM 檢查除了整體，也對主要 segment（裝置、新舊客、渠道）各做一次。

## HARKing / Segment Fishing 禁令

HARKing = Hypothesizing After the Results are Known：先看到結果，再回頭把它包裝成「我們本來就假設」。

Segment fishing 是它的常見形態：primary 不顯著，於是切 20 個 segment 找到「35–44 歲、Android、自然流量」顯著 → 宣稱對該族群有效。20 個 segment 各用 α = 0.05，期望值上本來就會有一個假顯著，這個「發現」極可能是雜訊。

規則：

- **事前聲明的 segment**（建議最多 3–5 個，每個附「為什麼預期有差異」的理由）：可以判讀，但要配多重比較校正或明確標註探索性。
- **事後發現的 segment**：一律標記「待驗證假設」，寫成新的假設卡 → 對該 segment 開新實驗。驗證通過才能行動。
- 報告用語紀律：事後 segment 只能寫「觀察到……值得後續驗證」，禁用「證實」「有效」。

## 何時不該跑 A/B

| 情境 | 原因 | 替代方案 |
|---|---|---|
| 效果量明顯巨大（修壞掉的結帳、致命 bug） | 不需要統計就看得出來，跑實驗是浪費 | 直接修，用監控與 pre-post 確認 |
| 倫理或公平性問題 | 定價測試尤其危險：同時段同產品對不同人收不同價，有公平性與法規風險 | 定價改用不同市場、時段、或包裝（方案組合）做 quasi-experiment |
| 樣本永遠不夠（B2B 月成交 20 筆） | 速查表上的樣本數這輩子湊不齊 | 質化驗證、pre-post + 控制組、拉長觀察的 cohort 比較 |
| 一次性事件（記者會、年度大促主視覺、超級盃廣告） | 沒有第二次可以分組 | 事前用小規模 pre-test（問卷、focus group），事後用時間序列歸因 |
| 效果回饋週期太長（LTV、年續約） | 等一年才能判讀，機會成本爆炸 | 已驗證的 proxy 指標 + 長期 holdback 組追蹤 |

## 替代驗證方案速覽

- **Pre-post + 控制組（difference-in-differences）**：改動只上部分市場/門市，比較前後差值的差。成立前提是「平行趨勢」——沒有改動時兩群走勢本來就平行。有大盤事件同時打到兩群不同力道時會失真。
- **Geo split**：以地區為分流單位。適合無法 user 級分流的媒體投放。地區數少時統計力弱，考慮 synthetic control。
- **Switchback**：同一單位按時段輪替開關。適合有網路效應、兩側市場（外送、叫車）的場景。時段間有殘留效應（carryover）時要留 washout 間隔。
- **Painted door / fake door**：功能還沒做，先放入口量點擊測需求。要處理好使用者預期（點了要給合理回饋），且只驗證「興趣」不驗證「留存」。
- **質化驗證**：5 人左右的 usability test 就能抓出大部分重大可用性問題。驗證「為什麼」很強，驗證「多少」很弱——與量化互補，不互替。

## 實驗登記模板

每個實驗結束（不論勝負或 inconclusive）填一份，存團隊共用位置：

```markdown
## EXP-{編號}: {實驗名稱}
- 期間：{start} ~ {end}（計畫 {n} 天 / 實際 {n} 天）
- Owner：
- 假設卡：因為觀察到…，我們相信…會導致…，可由…驗證
- 設計：分流單位 / 分組比例 / 觸發條件 / 排除規則 / 平台
- 樣本：計畫每組 {n}（baseline {x}%、MDE {y}%、α、power）/ 實際每組 {n}
- 健檢：SRM {pass/fail, p 值}；AA test {是否 / 結果}
- Primary 結果：效果量 {x}%（95% CI [{a}, {b}]）、p = {..}
- Guardrail：{各項變化與是否越線}
- Segment 觀察（事後、待驗證）：
- 決策：ship / no-ship / 重測 / 延長，理由：
- 學習：{對假設、對用戶、對流程各學到什麼}
- 後續行動：{衍生的新假設卡、要修的工具問題}
- 連結：{dashboard、分析 notebook、設計稿}
```

登記的紀律：

- 負結果與 inconclusive 一樣要寫——它們防止半年後重測同一個假設。
- 每季回顧勝率。業界常態是 10–30% 的實驗顯著正向；勝率接近 100% 通常代表 MDE 設太保守、在 peeking、或只登記贏的。
